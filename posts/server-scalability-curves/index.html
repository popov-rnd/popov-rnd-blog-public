Server scalability curves: a method of systems performance analysis

"About one practical method to assess and reason about a server performance under load"

Server scalability curves: a method of systems performance analysis
About one practical method to assess and reason about a server performance under load
performance, engineering
/_astro/cover.3Q4ZccIL.webp

Recently, I needed to analyse how several backend architectures behave under load when running on the same hardware.

The objective was to understand their maximal throughput, optimal concurrency levels, and performance at high load and during overload.

During my preliminary research in performance engineering, I discovered the concept of server scalability curves - a practical method for analysing and visualising system dynamics as load increases.

While the term itself is not canonical in the dedicated literature, it describes the idea very clearly.

I first encountered it in a blog post by Doug Hood (Oracle blog, 2025) titled “Server Scalability Curves – a Missing Law of Computing” (link at the bottom).

Below I summarise the essence of this method as a controlled approach for analysing system capacity under fixed conditions, and outline how these insights can be used for systems comparison and production tuning.

The problem: simplified benchmarks are misleading#

In practice, evaluating system performance under load is not straightforward.

Most engineers rely on benchmarking, yet many benchmark articles across the Internet follow a simplified pattern and present results in one of 2 misleading ways.

Approach #1#

Many benchmarks report results in the form: “System X processed 1 million requests in N seconds.”

From this, throughput is derived by simply dividing the number of completed requests by the total execution time.

While this produces a clean number, it hides the actual load conditions during the test.

The benchmark doesn’t show how many concurrent clients were used (and even if reported, whether that number was optimal?), how close the system was to its limits, or how response times changed during execution.

At a given concurrency level, one system may already be near saturation while another is still in its linear scaling region, which can distort comparisons between them.

Instead, I would want to see the “full picture” and compare maximal possible throughput of 2 or more systems being evaluated.

Approach #2#

Another common approach is to gradually increase the offered load against a target system until a latency constraint is violated, for example stopping the test once p95 or p99 latency exceeds a defined SLO (Service Level Objective), such as N ms.

This method is more realistic because it focuses on service quality rather than reporting a single raw throughput number.

By doing so, we can report the sustainable throughput that the system can process while meeting the latency objective.

However, this approach often starts from an arbitrary load level. So the system’s true capacity remains unknown.

It also doesn’t show how throughput evolves across concurrency levels, making it hard to reason about or tune internal params such as thread and connection pools.

Ideally, I’d also want to know the optimal concurrency level so upstream services can size these resources to align with downstream capacity.

Server Scalability Curves#

To better understand how a system behaves under load, we need to observe it at a more fundamental level, eventually introducing such terms as system capacity (maximum throughput) and optimal concurrency.

Instead of relying on isolated benchmark numbers, we should examine how key metrics evolve as pressure on the system increases.

As Srinath Perera highlights:

Performance is a dynamic measurement, unlike static server capacity measurements (e.g., CPU processing speed, memory size). Latency and throughput are strongly influenced by concurrency and work unit size.

This is where server scalability curves become useful as they allow us to visualize how throughput, latency and concurrency relate to each other as load grows.

Before proceeding, let me provide the definitions to the key concepts used throughout this article:

  • Throughput: the number of requests processed per unit of time, typically measured in requests per second (RPS).
  • Latency: the time required to complete a request, usually measured in milliseconds and often reported using percentiles such as p50 or p99 (meaning 99% of requests complete faster than that value).
  • Concurrency: the number of requests being processed by the server simultaneously.

Universal shape pattern#

In practice, most server systems - regardless of the technology stack, whether implemented in Java, Go, Node.js, Python or others - exhibit a recognizable performance pattern as load increases.

As Doug Hood observes:

“In over 30 years of computing, I have never seen a server system that does not fit within the model of Server Scalability Curves.”

When throughput and latency are plotted against concurrency, this behavior typically forms characteristic curves.

As concurrency grows, the system typically passes through several distinct regions (zones), each corresponding to a different operating regime (see Fig. 1-2).

  • Linear region: throughput grows almost linearly while latency stays stable, up to the “knee” (see below), where gains begin to slow and latency starts to rise;
  • Saturation region: throughput stops increasing, and additional load mainly results in queuing and higher latency;
  • Degradation region: under further pressure, throughput may decline while latency increases sharply.

Understanding these regions is important because they reveal not only the maximum throughput the system can achieve, but also the range in which it operates efficiently.

As a matter of fact, the real-world servers rarely produce identical curves.

As Doug Hood notes, systems may differ in several important ways (given a fixed workload and fixed hardware/software configuration):

  • How close they come to linear scalability under light load;
  • How far that linear region extends (how scalable it is);
  • How long throughput remains stable under heavy load;
  • How rapidly performance degrades once the system becomes overloaded.

Method to draw the curves#

To produce a server scalability curve, the system must be tested under a specific workload and hardware configuration.

Each setup produces its own curve; even small changes (e.g. thread pools, memory, framework version) can shift it.

The experiment begins by gradually increasing the load applied to the server. This is done by increasing the number of concurrent requests or connections to the system, using one of the load generators of choice: Grafana k6, Gatling, Bombardier, etc.

Doug Hood suggests to vary concurrency in powers of 2 (for example 1, 2, 4, 8, 16, 32, 64, and so on). This allows the system behaviour to be explored quickly across a wide range of load levels.

At each step, the system is exercised “long enough” to measure throughput and latency. There is no universal fixed duration. The test simply needs to process a sufficiently large number of requests so that latency percentiles, especially p99, become statistically meaningful.

As concurrency increases, throughput initially grows almost linearly.

Eventually a point is reached where throughput stops increasing significantly.

This indicates that the system is approaching its capacity — the maximum throughput the system can sustain.

To determine this transition more precisely, additional measurements can be taken between the last linear point and the saturation region.

A binary search strategy can then be used to narrow down the transition point by repeatedly testing the midpoint between 2 previously measured load levels.

Once the start of the heavy-load region is identified, concurrency can be increased further until throughput begins to decrease. This marks the onset of the degradation region.

At this stage latency typically grows very rapidly due to queueing effects (see below), so testing deep into this region is rarely necessary.

Finally, the collected measurements are plotted to produce throughput-versus-concurrency and latency-versus-concurrency curves.

These curves reveal the characteristic regions discussed earlier and allow the system’s capacity and optimal operating range to be identified.

Thus, the goal of the experiment is to figure out where these regions begin and end.

Why scalability curves appear#

The characteristic shape of scalability curves is not accidental. It reflects how servers process work under limited resources.

Every system has finite capacity: CPU, RAM, network, and internal concurrency limits. Each request consumes some service time.

When the arrival rate is below this capacity, requests are processed immediately. Throughput grows almost linearly, and latency remains stable.

As load increases, the system approaches its limits. Requests begin to wait for resources such as CPU time, threads, or connection pools.

At full utilization, throughput stops increasing — the system is already working at maximum capacity. From this point onward, additional load only creates queues. These queues may form at different layers: network (socket level), application (thread scheduling), or downstream systems (database connection pools).

As queues grow, latency rises rapidly while throughput remains flat (plateaus) or may even decline.

This behavior subordinates directly to the Little’s Law, a fundamental result from queueing theory:

Concurrency (N) = Throughput (X) × Latency (R)

Little’s Law itself doesn’t describe saturation (plateau), but it implies that once throughput (X) stops increasing while concurrency (N) continues to grow, response time (R) must increase accordingly.

Saturation itself is determined by the fundamental physical limits and discussed in such canonical performance engineering books as Raj Jain’s The Art of Computer Systems Performance Analysis and Neil Gunther’s Guerrilla Capacity Planning. The latter provides mathematical tools to model and quantify system’s scalability (Universal Scalability Law) and definitely deserves a separate blog post article to overview!

Thus, the scalability curves don’t “invent” this systems behaviour; they simply expose what is already dictated by underlying physics.

Key metrics and interpretations of the curves#

Key metrics#

Once a scalability curve is constructed, several key metrics can be derived:

  • Capacity (Xmax): the maximum throughput. On the curve, this corresponds to the “plateau” where throughput no longer increases. In practice, systems rarely operate here; instead, the “knee” throughput (Xknee) is used as a more realistic baseline.

  • Knee: the point where adding more load no longer increases throughput much, but latency starts growing quickly. It marks the moment when the system begins to queue requests. (see Raj Jain book, p. 35).

  • Optimal concurrency (Noptimal): the concurrency level near the “knee”, where throughput is high and latency remains stable.

As noted earlier, these metrics should be interpreted relative to the specific workload and system configuration.

Key applications#

Below are several practical purposes of the curves, discussed by Doug Hood:

  • First, it allows engineers to determine the maximum throughput a system can achieve while maintaining acceptable latency. This helps identify a safe operating point for the system.

  • Second, scalability curves help estimate the infrastructure required to support a desired workload. If the expected load exceeds the capacity of a single server, the curve can be used to estimate how many instances may be needed when scaling horizontally.

  • Finally, scalability curves provide a useful baseline when comparing different system configurations. For example, they can be used to evaluate new hardware, different server implementations, or updated software versions under the same workload.

Thus, having a scalability curve for a given server, hardware, and workload makes it possible to reason about system capacity and scalability in a structured way.

How to interpret?#

When analysing server scalability curves, several interpretations are particularly important:

  • A higher “plateau” indicates greater maximum throughput (Xmax);
  • A later “knee” indicates that the system maintains near-linear scaling over a wider concurrency range;
  • A flatter latency curve before the “knee” reflects more stable response times under load;
  • A wider “plateau” region shows the system can sustain high throughput over a broader concurrency range;
  • A sharper drop after saturation indicates weaker overload resilience.

Taken together, these characteristics describe not only how fast a system can operate, but also how predictably it behaves as it approaches its limits. Systems with a later “knee” and smoother degradation generally remain more stable under real-world load conditions.

Closed and open workload models in performance testing#

Closed workload model#

Server scalability curves are typically obtained using so-called closed workload model.

In this model the number of concurrent clients is fixed, and each client sends a new request only after the previous one completes. As a result, the load applied to the system is controlled directly through the level of concurrency.

However, this model has an important limitation. Because new requests are only issued after previous ones complete, the system’s response time directly influences the rate at which new requests are generated.

If the server becomes slow under load, clients simply wait longer before sending the next request. As a result, the arrival rate of new requests decreases automatically. In Grafana Labs documentation this effect is described as coordinated omission.

Open workload model#

In many real-world systems, new requests continue to arrive even if previous ones have not completed.

This corresponds to an open workload model, where requests are driven by an arrival rate rather than a fixed number of concurrent clients.

Open models are therefore often considered more representative of real production traffic, especially for public services exposed to unpredictable user demand.

Both models are useful: closed models are well suited for analysing scalability behaviour, while open models are better for evaluating how systems behave under realistic traffic conditions.

Open vs closed model#

It should be noted that systems which appear to perform well under closed workload models may exhibit much worse behavior under open workload conditions.

In closed models, increasing latency reduces throughput automatically, which softens the visible effects of saturation. In contrast, open models continue to apply pressure, exposing queueing and latency collapse much more clearly.

Therefore, scalability curves derived from closed models should be interpreted as an upper bound of system capability rather than a directly sustainable production load.

Modern tooling#

Modern load-testing tools such as k6 and Gatling typically support both workload models through the concept of “virtual users”.

A “Virtual user (VU)” represents an independent client that repeatedly sends requests to the server.

In a closed workload model, the number of VUs is fixed. Each VU sends a new request only after the previous one has completed, which means the system load is limited by the number of concurrent users.

In an open workload model, requests are generated at a target arrival rate (for example a fixed number of requests per second). To maintain this rate, the load-testing tool dynamically creates additional VUs whenever existing ones become blocked or slow.

This allows engineers to test systems under controlled concurrency or under fixed arrival-rate workloads that more closely resemble real production traffic.

Two-step performance analysis approach#

Having discussed the limitations of existing benchmarking approaches and introduced the concept of Server Scalability Curves, we can outline a practical performance analysis procedure.

Given that the hardware environment and system configuration are fixed, system evaluation can be performed as a two-step process:

  1. Measure and draw the server scalability curves*. Their shape immediately reveals the key properties of the target setup:
    • Maximum throughput (Xmax),
    • Knee throughput(Xknee),
    • Optimal concurrency (Noptimal).

This provides 2 important insights: “what load level the system can potentially sustain” and “how to size internal thread and connection pools”.

The latter is particularly useful for capacity alignment between services: upstream connection pools shouldn’t exceed the downstream system’s optimal concurrency (Noptimal), otherwise excess load only translates into queueing and latency amplification.

  1. Test the system under a more realistic production scenario using an open workload model, operating roughly in the range of 0.5Xmax to Xmax with a reasonable step size. At this stage latency distributions such as p50, p95, and p99, as well as maximal observed latencies, can be captured and reported. This step validates the behaviour of the system under production-like traffic conditions.

Together, these 2 steps reveal both the intrinsic capacity of the system and its latency behaviour under realistic load, providing a more complete basis for performance analysis.

During both phases it’s also useful to observe CPU utilisation, RAM consumption and connection pool usage (if any downstream dependency is present) to identify the dominant constraint and explain where scalability breaks down.

In my experiments, this method turned out to be significantly more informative than traditional “single-number benchmarks”, as it exposes system behavior across the entire load spectrum rather than at an arbitrary point(s).

In this article, I intentionally isolate the method from experimental results; concrete measurements and system comparisons will be presented in the next part of this series.

Conclusion#

After running numerous experiments mainly with Java-based stack, the same patterns consistently appeared:

  • Scalability curves reliably follow the same shape: linear growth -> “knee” -> saturation;
  • The position of the “knee” of the curve heavily depends on the workload type: CPU- or IO-bound;
    • For CPU-bound workload the “knee” appears much earlier, as concurrency cannot exceed available CPU capacity; in my experiments with 1–2 CPU core containers, it occurs at ~4–16 concurrency levels;
    • For IO-bound workloads the “knee” appears order of magnitude later, as requests spend much of their time waiting on external services, allowing higher concurrency; it depends on the slowest downstream service latency and may occur at ~256–1024 concurrency or beyond.
  • In open workload model conditions, throughput near the “knee” is rarely sustainable: systems typically operate at ~0.6–0.9·Xmax to maintain acceptable latency.

In essence, system behavior is not defined by peak throughput, but by where the “knee” occurs and how the system behaves around it.

Systems whose curves reach the “knee” later generally sustain higher load under open workloads before latency degradation begins, making the “knee” position a practical indicator for comparing systems under identical conditions.

These observations are consistent with earlier experimental reports (see LinkedIn posts 1, 2), where similar patterns can be observed across different backend architectures.

However, this method addresses only one aspect of performance analysis and doesn’t cover full benchmarking or performance tuning, which are outside the scope of this article.
