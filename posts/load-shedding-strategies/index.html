Load-shedding strategies for Spring Boot on Tomcat

"This article examines how load shedding can be implemented in Spring Boot services running on Tomcat, from the executor level to application-level admission control"

Load shedding is a familiar reliability concept, actively discussed in AWS, Google SRE, Netflix, and other production-engineering materials.

Java engineers are also familiar with the individual concurrency-control mechanisms: thread pools, semaphores, rate limiters, and bounded connection pools.

What is less obvious is how to combine them into a coherent load-shedding strategy for Spring Boot services running on Tomcat.

This article addresses that gap by comparing both Tomcat’s platform thread pools (PT) and virtual thread per request (VT) execution models, and evaluating how well each supports modern load shedding.

Before presenting the final strategy, we examine the relevant Java implementations of the underlying concurrency primitives: their operational costs, rejection points and waiting behavior.

Let us begin#

Let’s start by providing a definition of “load shedding”, synthesized from AWS and Google SRE materials (1, 2):

“Load shedding” is the deliberate rejection of excess work when a system approaches or exceeds its safe capacity, so that the accepted workload can continue to be processed with predictable latency and availability.

The term itself comes from electrical power systems field: when electricity demand becomes too high, some load is intentionally disconnected to prevent the whole grid from failing.

In software systems, the goal is similar: reject part of the offered load early enough to preserve goodput — the useful part of throughput, meaning successful responses completed within the client’s deadline.

Without load shedding, an overloaded server may continue accepting more work, but spend increasing capacity on queued, slow, retried, or already-doomed requests.

The broad strategy can be stated simply:

Reject excess requests as early and as cheaply as possible, preferably with a clear HTTP 503 Service Unavailable response, ideally before they enter deeper application logic and increase CPU, RAM, and GC pressure.

Our task is to turn this general principle into a concrete load-shedding strategy for heavily loaded Spring Boot services running on embedded Tomcat.

Concurrency-control primitives#

Exploring concurrency-control “primitives”, such as thread pools and semaphores, in-depth is not the main objective of this article.

Yet, because these primitives are used to implement real load-shedding boundaries, so it is useful to briefly recall how they behave under pressure: how they limit concurrency, where excess work waits, and how it eventually fails.

Thread pools: sized queuing#

Java thread pools implement the ExecutorService abstraction: they accept submitted tasks and execute them using a bounded set of reusable worker threads.

For load-shedding purposes, a thread pool can be seen as an concurrency boundary: it allows up to N tasks to run concurrently and, with a bounded queue, up to Q additional tasks to wait. This creates a clear population-based overload boundary: once both the worker population and the queue population are full, new submissions are rejected according to the configured rejection policy.

This model also should not be applied to virtual threads (VTs): Oracle explicitly discourages pooling VTs to limit concurrency; they should normally be created per task, while concurrency is bounded separately with mechanisms such as semaphores.

Thread pools are therefore a natural fit for application servers that execute requests on platform threads (PTs), giving the server one simple mechanism to control both concurrency and backlog.

Semaphores: timout-bounded waiting#

In Java, Semaphore is an implementation of synchronization primitive, which controls access to a critical section through a fixed number of permits.

When no permit is available, the caller may block indefinitely, fail immediately, or wait up to a timeout, depending on the acquisition method used.

Internally, Java’s Semaphore is built on AQS synchronizer framework, where excess callers may wait in an internal linked queue with approximately FIFO ordering. With timed acquisition, each waiter is bounded by its own timeout: it either obtains a permit or leaves the queue when the timeout expires. This creates a time-based overload boundary: at most N callers enter the protected path, while others may optionally wait for no longer than T.

Semaphores work with both PTs and VTs, but they become especially important with VTs, where concurrency is no longer naturally limited by thread-pool size.

Load-shedding strategies#

The previous sections describe concurrency-limit primitives independently.

The next step is to combine them into a general load-shedding strategy for Spring Boot services running on Tomcat.

Instead of relying on one admission point, the strategy uses 2 complementary admission boundaries:

  • Global boundary, NG | Server level: protecting the application as a whole;
  • Local boundaries, Ni | Application level: isolating individual critical API paths or dependencies.

The global boundary solves 2 practical problems:

  • It bounds the aggregate concurrency of the whole server, including paths without local protection;
  • It rejects overload earlier, before requests enter deeper framework infrastructure and application logic.

Similar ideas are described by Luis Liguori in the AWS article “Using dependency isolation to contain concurrency overload”.

The next sections apply these ideas specifically to Apache Tomcat’s execution models: PT-based (StandardThreadExecutor) and VT-based (StandardVirtualThreadExecutor).

Global boundary#

Option #1: PT-based executor#

For an application powered by the PT-based executor (StandardThreadExecutor), the thread-pool size maxThreads = N naturally limits global request-processing concurrency to at most N active platform threads, see Fig 2a.

Tomcat uses a customized variant of ThreadPoolExecutor: it eagerly prestarts minSpareThreads core workers, creates additional workers as submitted demand exceeds the current pool instead of filling the TaskQueue immediately, grows up to maxThreads, and later retires idle workers above minSpareThreads after maxIdleTime, see doc.

It lets Tomcat adjust the pool size to actual traffic, reducing idle platform-thread overhead while still scaling up to maxThreads under load.

The TaskQueue should be sized from the maximum acceptable queueing delay: a small Q can absorb brief traffic spikes, while an effectively queue-less configuration (maxQueueSize = 1) is preferable when latency must remain predictable and immediate rejection is acceptable.

Once all N threads are busy and the task queue Q is full, the executor rejects new request tasks before they enter deeper application processing. Because rejection occurs below HTTP layer, the application cannot return a clean response HTTP 503 Service Unavailable; clients may see a closed connection instead.

The upstream infrastructure: a reverse proxy, ingress controller, or load balancer, can still produce some explicit error code like 502 Bad Gateway, yet the exact failure reason cannot be interpreted.

This executor is suitable for low-to-moderate global concurrency NG, where the cost of maintaining N platform threads remains acceptable and explicit overload response is not a strict requirement.

Option #2: VT-based executor#

For an application powered by the VT-based executor (StandardVirtualThreadExecutor), the executor creates a new VT for each submitted request task, see Fig. 2b.

Unlike a PT-based executor, it provides neither a configurable worker pool nor a bounded task queue. Consequently, the executor itself does not impose a meaningful global concurrency boundary which must be introduced separately.

Apache Tomcat provides Valves, Tomcat-specific interceptors in Catalina’s container pipeline. They execute before the web application’s Servlet filter chain, enabling requests to be inspected or rejected before entering deeper application processing.

SemaphoreValve uses this mechanism to add concurrency admission control: with concurrency = N permits, at most N requests are allowed to continue concurrently beyond the valve. Valves belong to the Tomcat container itself and can be attached at the Engine, Host, or Context level.

By default, SemaphoreValve uses block = true, so excess requests wait for a permit rather than being shed. With block = false and highConcurrencyStatus = 503, it becomes a server-level load-shedding mechanism: requests beyond N concurrent permits are rejected immediately with HTTP 503 Service Unavailable.

Because rejection occurs inside Tomcat rather than at the executor boundary, it carries slightly more per-request overhead. However, the explicit HTTP 503 response provides a clearer overload signal and more predictable load-shedding behavior.

This is one possible impl. of a global concurrency boundary NG, not the only one. Alternatives include 3d-party concurrency-control libraries and custom Servlet filters, with different integration points and rejection costs, though those alternatives act later in request processing.

Overall, this executor is suitable for highly concurrent applications with many in-flight VTs, where global application concurrency still requires an explicit bound.

Local boundaries#

A global concurrency limit cannot distinguish cheap request paths from expensive ones.

Local boundaries Fig. 3 add that distinction by protecting selected critical paths separately.

Let’s make a separation between different kinds of downstream dependencies:

  • Controlled with known and testable capacity;
  • Uncontrolled or with highly variable capacity.

Controlled dependencies#

For an internal dependency with known and testable capacity, Ni can be set from the maximum concurrency it sustains without unacceptable latency or errors. Ti should follow the path’s latency budget and permit only short, measured bursts. Total concurrency should generally satisfy N<sub>G</sub> > sum(N<sub>i</sub>).

In CRUD-heavy applications, the database connection pool, e.g., HikariCP, acts much like semaphore for database access. Its configurable maximumPoolSize limits the number of connections that can be used concurrently, while connectionTimeout controls how long callers wait when the pool is exhausted.

Unlike general semaphore, however, this boundary applies specifically to concurrent use of DBMS connections. Even so, semaphores can be added to expensive paths, such as reports or complex searches, so they do not consume too many connections.

Uncontrolled dependencies#

External APIs are often different because they are limited by request rate rather than by a known safe concurrency. In that case, a rate limiter controls how fast requests are sent, while semaphore can still limit how many calls remain in flight.

For a rate-limited external API, Little’s Law estimates the expected concurrency as N ≈ λW, where λ is the allowed request rate and W is the average downstream latency. This value can be used as the initial semaphore limit, with some headroom for normal latency variation and later validation under load.

For shared or autoscaled dependencies, safe concurrency may change over time. Adaptive algorithms that use latency feedback, including Vegas, Gradient2 or similar, to adjust the allowed in-flight concurrency dynamically can be used instead.

Overall, choosing the right limits is part of capacity planning and performance testing, and it can be challenging when the application has many paths, changing traffic, or downstream systems with unstable capacity.

Strategy formulated#

This strategy assumes both limits NG and Ni have already been measured or estimated.

Given these prerequisites, the overall load-shedding strategy can be outlined as follows:

  • Configure the global boundary by executor:
    • PT-based | StandardThreadExecutor: set maxThreads = NG and keep maxQueueSize = Q small; use Q = 1 for near-immediate rejection by closing connection.
    • VT-based | StandardVirtualThreadExecutor: treat the executor as unbounded and configure a context-level SemaphoreValve with concurrency = NG, block = false, and highConcurrencyStatus = 503.
  • Configure local boundaries: restrict concurrency around critical paths and dependencies as early as possible, preferably at the beginning of servlet filter chain. For dependencies with:
    • Strict capacity: place semaphores with limits Ni in front of critical sections; prefer fail-fast acquisition to avoid local queueing;
    • Elastic capacity: an adaptive algorithm can be used instead, but the adaptation should remain bounded by measured limits: Ni,min ≤ Ni ≤ Ni,max.

That said, with a VT-based executor, local boundaries alone may be enough when all meaningful work passes through protected paths; if not, and no global boundary is configured, excess traffic can bypass admission control, queue deeper in the app, increase latency, or create enough VTs and associated request object trees to exhaust heap.

As we can see, both executor models can implement the same global + local admission strategy, but differ in overload signaling:

  • VT-based: explicit HTTP 503 from the global boundary;
  • PT-based: closed connection, often surfaced upstream as HTTP 502.

Why not use a SemaphoreValve with the PT-based executor as well? It is possible, but under heavy overload, executor may exhaust its worker threads and requests may be rejected before reaching the valve, preventing it from returning a clear response code.

Final thoughts#

Load shedding requires a shift in engineering mindset: production services must be designed not only for functional correctness under normal conditions, but also for defined behavior under overload.

Before release, teams should explicitly define the overload failure mode: where excess work waits, where it is rejected, and how upstream callers are expected to react.

Instance-level load shedding is not a panacea. Even rejected requests consume CPU, heap, and GC activity while they are accepted, routed, and returned. Under sustained overload, this overhead can still reduce goodput. Production systems therefore also require upstream protection, such as DDoS mitigation, API key rate limits, and retry budgets with exponential backoff. For legitimate traffic, autoscaling should normally react to leading saturation signals before load shedding dominates.

After looking more closely at both Tomcat’s executor models, the choice becomes more nuanced. Even when the service’s “useful-concurrency”, expressed here as NG, is relatively small, a VT-based executor can be a better fit when overload is an expected operating condition. Paired with explicit global semaphore, it can preserve goodput by rejecting excess requests with explicit HTTP 503 instead of hiding overload in connection failures or growing executor queues.
