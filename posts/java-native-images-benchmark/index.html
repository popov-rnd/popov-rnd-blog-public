Java native images: static and performance analysis
This benchmark compares 3 native Java executables built with Spring Boot, Quarkus and Micronaut, each configured with a minimal setup exposing only the web layer.
graalvm, performance, benchmark

As of 2026, the Java ecosystem has matured to support building native executables via ahead-of-time (AOT) compilation. Today, all major Java frameworks such as Spring Boot, Quarkus and Micronaut provide native image support out of the box.

However, there are still relatively few trusted benchmarks that compare the native images produced by these frameworks in an unbiased and reproducible way.

In this article, I aim to address that gap by conducting my independent evaluation.

The analysis is structured in 2 parts:

  • Static characteristics: startup time, memory footprint (RSS), executable size, etc.;
  • Runtime characteristics: max throughput and latency behavior under load.

Set-up#

To ensure objectivity, I created 3 separate Java projects in my IDE based on Spring Boot, Quarkus, and Micronaut (GitHub link will be attached at the end).

Each project includes only the minimal set of dependencies required to expose the web layer (HTTP server, JSON binding, validation, etc.).

Dependency selection was framework-driven: some frameworks provide bundled starters, others require manual composition. The intent was to compare metrics in realistic “minimal viable” setups, not a synthetic dependency set.

I deliberately did not attempt to normalize the underlying web server across frameworks, instead selecting the default, recommended stack for each framework.

As such:

  • Spring Boot exposes Tomcat (servlet-based stack);
  • Quarkus exposes Netty via Vert.x;
  • Micronaut exposes Netty.

Each framework provides official guidance for building native images:

  • Spring Boot via native build tools or cloud native buildpacks;
  • Quarkus via local build or container-based native build;
  • Micronaut via local build or containerised build;

Each framework typically provides 2 approaches for building native executables using:

  • locally installed GraalVM distribution;
  • a containerized build process that does not require a local GraalVM installation.

For this experiment, I used the local build approach and the same GraalVM version across all frameworks to ensure a normalized and comparable setup.

In each case, the officially supported Maven or Gradle plugin provided by the framework was used to drive the native image build process.

Default GraalVM optimization level -O2 was used, as it targets runtime performance. Default GraalVM machine architecture flag was used -march=compatibility, as it targets for best portability. Default GraalVM garbage collector impl. being Serial GC was used, the only supported by CE (Community Edition) distribution.

Hardware#

I conducted all measurements on my local Linux Mint (kernel 5.15.0) PC with the following configuration:

  • CPU Intel Core i7, 8x2.9GHz
  • SSD NVMe PCIe 4.0 x4, 500GB
  • RAM DDR4-2666, 32Gb

Software#

For this benchmark, I used locally installed via SDKMAN GraalVM JDK 25 CE (Community Edition, except for Quarkus I used Mandrel).

Additional software included (latest versions as of Jan 2026):

  • Docker engine v. 29.1.3;
  • Spring Boot v. 4.0;
  • Quarkus v. 3.30;
  • Micronaut v. 4.6;
  • GraalVM JDK CE or Mandrel v. 25;
  • Grafana k6, v. 1.6

Static characteristics#

I measured the following characteristics:

  • Startup time -> taken after first http 200 response (not relying on, framework logs “Started in _ s” messages).
  • Memory usage -> from standard RSS Unix tool, ps -p $MY_PID -o rss.
  • Artifact size -> from ls -lh or direct filesystem metadata.
  • Build time -> taken from Maven’s own build output (e.g., [INFO] BUILD SUCCESS in _ s).

Method#

For collecting metrics I used .sh script, similar to the one featured in the presentation “Bootiful GraalVM by Thomas Wuerthinger / Josh Long @ Spring I/O 2025”.

Every of 3 apps contained empty GET / endpoint for measuring the moment when the first response after start-up is received.

Results#

Startup time (Fig. 1.1). Startup was defined as the moment the first HTTP 200 response was received, polled every 1 ms (no reliance on framework logs). For both cold and warm starts, Quarkus was consistently the fastest.

Memory footprint (Fig. 1.2). Measured immediately after the first successful response via standard Unix RSS tools. Here, Quarkus consumed the least memory before any load.

Native executable size (Fig. 1.3). The final native artifacts produced by framework building plugins showed Quarkus as the most compact.

Additional observations during the test:

  • Native compilation time ranged from 59-69 sec across frameworks, and the process fully saturated CPU cores.
  • Shared memory was negligible and similar across frameworks, so RSS closely matched private memory per instance.
  • Readiness reported by the frameworks occurred 2-3× earlier than the apps were actually ready to serve HTTP requests.

Quarkus native showed the leanest overall static profile across startup time, memory usage, and binary size. Still, the real takeaway isn’t the exact numbers but the general scale: all frameworks ended up in a similar performance range typical for modern native Java apps.

These results help set realistic expectations rather than define hard, fixed values.
