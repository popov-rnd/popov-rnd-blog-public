Spring Boot | Tomcat request execution models

Spring Boot | Tomcat request execution models
This article explains how Spring Boot with embedded Tomcat executes blocking HTTP requests using either a classic worker thread pool or virtual threads. It compares scalability, resource usage, overload behavior, and offers a practical heuristic for choosing the right execution model.

Virtual threads in Java are often discussed in isolation, detached from the realities of enterprise application development.

But their practical value only becomes clear when placed inside the request execution model most Java teams actually use: Spring Boot running on a Servlet container such as Apache Tomcat.

This article compares how the same blocking programming model behaves when incoming requests are executed by a worker platform thread (PT) from a bounded thread pool versus newer virtual threads (VT).

The goal is to build a mental model of how Spring Boot and Tomcat execute blocking requests, and to clarify the differences in scalability, resource usage, failure modes, and operational trade-offs.

In the end, this should help answer a real question: when should you stay with PT-per-request, and when does VT-per-request actually make sense?

But first, let’s step back and look at the broader evolution of the Spring stack.

Evolution#

The milestones below cover only Spring MVC (servlet-stack), whereas Spring WebFlux (reactive path) is out of scope for this article.

Milestone #1: platform thread pools, ~2004–today#

In the classic Spring MVC/Tomcat model, request execution is built around a platform thread pool managed by the Servlet container.

Each HTTP request is handled by an OS-backed worker thread that carries the request through the Servlet pipeline, Spring MVC, application code, and any blocking I/O.

This model became the industry default because it is simple, productive, and maps naturally to imperative Java code.

However, under high-concurrency I/O-bound workloads, many requests could block and keep worker threads busy, forcing teams to use larger pools, accept higher memory usage, or queue/reject more requests.

Milestone #2: async servlet processing, ~2009/2010–today#

Servlet 3.0 (~2009) async processing introduced a way to start the request on a container worker thread, release that thread while the response is not ready, and complete the response later.

Spring MVC exposed this model through a dedicated API for long-running requests, delayed responses, and streaming responses.

This usually did not remove blocking; it relocated it to another PT-based executor and introduced a more asynchronous programming style.

So this model improved utilization of Tomcat’s worker threads, but it did not remove platform-thread pool management and remained a specialized tool.

Milestone #3: virtual threads era, ~2023–today#

In June 2023, Tomcat added support for VT-per-request execution through a dedicated executor, and Spring Boot made this model easy to enable from configuration.

VTs changed the trade-off by making the classic thread-per-request model much cheaper for I/O-heavy workloads.

However, this also changes the failure mode: if the application can accept far more concurrent requests, then a load shedding strategy becomes mandatory rather than optional.

For a fuller history of Tomcat’s own internal evolution, see Mark Thomas’s presentation.

Let us begin#

Today, PT-per-request and VT-per-request coexist as 2 blocking request execution models for Java enterprise workloads.

Each has conditions where it fits best, so the rest of this article compares them directly.

But first, let’s briefly review how Spring Boot starts and configures Apache Tomcat.

How Spring Boot bootstraps Tomcat#

Spring Boot initializes Tomcat programmatically during application startup, running it as an embedded web server inside the same JVM process as your application.

Tomcat itself remains an independently developed and maintained project. Spring Boot just brings it in through a dedicated starter and, during startup, creates and configures a Tomcat instance through its server factory, GitHub.

Tomcat-specific settings are exposed through Spring Boot’s externalized configuration under the server.tomcat.* prefix and applied through dedicated customizers.

During startup, Spring Boot also registers its DispatcherServlet with the Servlet container, after which Tomcat owns the incoming HTTP request path until the request reaches Spring MVC and application code.

Since Spring Boot 3.2, VT-per-request execution can be enabled with a single flag:

spring:
  threads:
    virtual:
      enabled: ${VT_ENABLED:false}

Internally, Spring Boot uses this flag to configure Tomcat’s request-processing executor to use virtual threads.

Importantly, this flag is not Tomcat-specific. It is an application-wide Spring Boot setting that also affects framework-managed task execution, scheduling, and asynchronous processing. This design choice is discussed by Brian Clozel in this GitHub comment.

Overall, Spring Boot lets the application choose the execution model through configuration, and then adapts embedded Tomcat accordingly.

How Tomcat is designed internally#

Apache Tomcat was designed from the beginning primarily as a Servlet/JSP container, with built-in HTTP handling implemented by its Connectors.

For this article, it is enough to view Tomcat as 2 large logical parts:

  • Connector / Network side: accepts TCP connections, watches sockets, and schedules socket-processing work;
  • Container / Catalina side: runs the Servlet pipeline and eventually reaches framework and application code.

I’ll focus on the Connector, where the distinction between PT-per-request and VT-per-request begins.

Tomcat provides 2 main implementations: the default, recommended, and best-documented NIO connector, and NIO2. They expose nearly identical behaviour, differing mainly in their low-level socket I/O impl.

The official Tomcat UML-sequence diagram explains how request processing is performed inside the NIO Connector.

  • Acceptor accepts new socket connections (up to maxConnections);
  • Poller detects socket readiness and submits processing work to an Executor;
  • Executor runs the submitted task on either a platform thread or a virtual thread;
  • Coyote is the protocol engine that handles HTTP/1.1 and HTTP/2 mechanics.

When Acceptor has already reached the connections limit, new connection attempts may wait in the OS backlog queue, whose requested size is controlled by acceptCount.

In both Connectors, a ready socket is wrapped in a lightweight SocketProcessor (NIO, NIO2): an internal Runnable submitted to the Executor to continue processing through Coyote and, eventually, the Servlet container.

The next sections look at both Executor implementations separately.

Model #1: PT-per-request#

Fig. 1 shows how Apache Tomcat executes blocking requests using a worker platform thread pool.

In this model, a normal blocking request occupies one PT for the whole request-processing path.

In this case, the SocketProcessor task is submitted to StandardThreadExecutor impl., which is backed by a worker thread pool and Tomcat’s specially designed TaskQueue.

When all current workers are busy and the pool can still grow, Tomcat creates additional PTs. After the pool reaches its configured limit (maxThreads = 200 by default), new tasks are queued instead.

When load decreases, Tomcat retains at least minSpareThreads and retires extra idle worker PTs after the configured idle timeout.

As a matter of fact, the TaskQueue is virtually unbounded by default (maxQueueSize = Integer.MAX_VALUE) and doesn’t have any configurable timeout that expires queued tasks.

This means that a large number of tasks may accumulate there, leading to growing latency, memory pressure, and slow recovery from overload.

If this queue is bounded and becomes full, submitted work is rejected by default, resulting in forcing closing the TCP connection.

Interestingly, Spring Boot started to expose a direct config parameter for TaskQueue only relatively recently, after issue submitted by Moritz Halbritter (~ 2023).

This highlighted the importance of explicitly bounding the queue based on a particular workload.

Overall, this model is simple, mature, and still a good fit for many Spring Boot enterprise applications.

StandardThreadExecutor: trade-off

Its main advantages are what we get out of the box:

  • The worker thread pool provides a clear concurrency limit;
  • Queueing can absorb short overload spikes instead of rejecting too early;
  • The model is battle-tested for mixed or CPU-heavy workloads with moderate concurrency.

Its main limitation comes from the 1:1 mapping between platform threads and native OS threads:

  • High I/O concurrency translates directly into more native stack memory, more OS-visible threads, and higher scheduling overhead.

Model #2: VT-per-request#

Fig. 2 shows how Apache Tomcat executes blocking requests using virtual threads.

In this model, a normal blocking request occupies one VT for the whole request-processing path, instead of holding a worker PT forever.

In this case, Tomcat submits SocketProcessor object to StandardVirtualThreadExecutor impl., which is much simpler than StandardThreadExecutor.

Every submitted socket-processing task is executed on a newly created VT, virtually removing natural thread pool concurrency cap and any explicit queueing.

Tomcat adds no extra concurrency limit here: no configurable semaphore, no request cap. Restricting concurrency becomes the application developer’s responsibility.

A VT is still a Java thread, but it is much cheaper to keep blocked; VTs use small, dynamically growing heap-stored stacks with an initial size measured in few KB.

VTs are scheduled by the JVM onto a smaller set of PTs called carrier threads, which in turn are still mapped 1:1 to OS threads.

In other words, Java uses an M:N model here: many VTs (M) are multiplexed over fewer carrier PTs (N), M >> N.

As stated in JEP 444, the carrier pool is a dedicated ForkJoinPool, with pool size being equal to the number of available CPU cores.

When a VT blocks on I/O, the JVM can park and unmount it from the carrier, allowing that carrier to mount and run another VT.

This makes this model more adapted to highly-concurrent applications with many blocked in-flight requests.

At the same time, this introduces an intermediate scheduler that parks and resumes VTs, which at scale may add extra churn for CPU-bound workloads.

StandardVirtualThreadExecutor: trade-off

Its main advantages come from VTs advantages:

  • Cheaper I/O concurrency, because blocked requests do not keep a dedicated OS thread and native stack memory while waiting.

Its main limitation (as of Java 25) is:

  • Pinning can reduce scalability when a VT cannot unmount from its carrier (native method or a foreign function);

How do models behave under pressure?#

To understand the practical difference between the 2 execution models, it is not enough to look only at the request flow.

The real difference becomes visible under load: when concurrency grows, capacity is reached, and the system starts to resist.

For this reason, I will model the behavior of PT-per-request and VT-per-request execution in 3 situations, as shown in Fig. 3:

  • Normal, low concurrency, N < 10;
  • Saturation, high concurrency, capacity is reached;
  • Degradation, extreme concurrency, N >> Nnorm

More explanation of these regions is available in this article.

PT-per-request execution under load#

Normal: the system has enough idle worker PTs, stable processing

Tomcat accepts the next TCP connection, waits for the Poller to confirm socket readiness, and submits a SocketProcessor task to the StandardThreadExecutor.

The executor can immediately hand the task to an available worker PT. Further processing runs on that PT along the full request path. On a blocking I/O call, the worker PT remains occupied by that request until the I/O completes.

The system is stable and clients receive responses with low latency.

Saturation: worker PTs become the limiting resource, queueing begins

At this load point, often around N ≈ maxThreads for blocking workloads, all worker PTs inside the StandardThreadExecutor are busy processing other requests.

New SocketProcessor tasks start spending more time in the executor’s TaskQueue until a worker PT becomes available. The larger the gap between the rate of incoming work and the rate of completed requests, the faster the queue grows.

This causes latency to increase super-linearly. Throughput no longer grows proportionally with concurrency and eventually reaches a plateau.

Clients or an intermediary such as a load balancer can still time out.

Degradation: the system moves from slow responses to failed connection attempts

Depending on the HTTP protocol version and connector configuration, the TaskQueue may become full and reject new tasks, causing TCP connections to close.

Alternatively, the Acceptor may reach its maxConnections limit forcing new connection attempts into the OS accept backlog up to the configured acceptCount.

This queue is relatively small, so once it becomes full, the OS may start refusing new TCP connections.

Exact behavior is OS- and config-specific, but the result is the same: new TCP connections are no longer established reliably.

Clients start seeing connection timeouts.

VT-per-request execution under load#

Normal: the system keeps issuing new VTs, stable processing

Tomcat accepts the next TCP connection, waits for the Poller to confirm socket readiness, and submits a SocketProcessor task to the StandardVirtualThreadExecutor.

The executor runs the task on a new VT. Further processing continues on that VT along the full request path. On a blocking I/O operation, JVM may park and unmount VT from carrier and mount back when response is ready.

The system is stable and operates normally with low latency.

Saturation: app-level concurrency limit is reached

At this load point, if an app-level concurrency limit, for example Semaphore-based, is set and reached, the system either rejects requests immediately, usually translating them into HTTP 429 responses, or makes requests wait — optionally with a timeout — for a permit, causing app-level queueing and increased latency.

In the absence of app-level concurrency restrictions, excessive request concurrency may instead overwhelm downstream services or cause queuing on connection pools.

This queuing is more memory-expensive here because each request has already entered the application and retains a larger object graph.

Client experience then depends on the downstream behavior: from increased latency and explicit errors to connection-pool exhaustion, for example reaching the maxConnections limit in HTTP/1.1 clients.

Degradation: the increased load is no longer efficiently translated into goodput

Assume an app-level concurrency limit is set. Load beyond the limit produces more rejected requests, usually translated into HTTP 429 responses. This protects the downstream path, but rejection is not free: exception-heavy or log-heavy rejection can still consume CPU and memory.

As the rejected share grows, goodput — successful 2xx responses — drops and the server enters the degradation regime. Some requests still complete successfully, while most excess clients receive immediate 429 responses after reaching the app.

Recovery is usually smoother than with uncontrolled overload because excess work is rejected near the app boundary instead of accumulating deeper in the system.

”Useful concurrency” as the selection criterion#

The load modeling shows that for both execution models a concurrency cap is mandatory in any production system with non-trivial load.

Ignoring this boundary does not create “unlimited scalability” but rather leads to unpredictable consequences under pressure, mentioned above.

Hence, the logical question is: “What is the right concurrency” level? Let me introduce and define this term:

Useful concurrency is the maximum number of in-flight requests the system can sustain without overwhelming downstream dependencies or the server instance itself.

Finding this level is a separate performance engineering task, that is grounded in Little’s Law, Server Scalability Curves, Universal Scalability Law and other practices.

Once this number is roughly understood, it can guide engineers on which model fits the target workload best, see Fig. 4.

The decision tree follows the intuition from JEP 444, which states that VTs are useful when:

  • The workload is not CPU-bound;

  • The concurrency level is > a few thousand.

The first practical boundary is Tomcat’s default PT worker pool size limit, N = 200, which is a reasonable default for many enterprise workloads.

The second is around N = 1000+, where the native stack cost of PTs can start to matter seriously in container memory budgets, adding extra ~136 MB–1 GB.

Between these values, both models are viable: VTs become attractive when requests mostly wait on I/O, while PTs remain preferable when requests mostly consume CPU.

At N >> 1000, VTs are bounded by hardware and downstream capacity, and begin to compete directly with event-loop-based solutions, which I will cover in the next articles.

Having said that, the final decision should still be validated with benchmarks on the target hardware and expected workload.

Conclusion#

Platform worker thread pools are not obsolete, and virtual threads do not provide “unlimited scalability”.

Virtual threads make blocked in-flight requests cheaper, but they do not remove real bottlenecks such as CPU, memory, connection pools, locks, or downstream business throttling.

Apache Tomcat supports both models through a dedicated Executor:

  • StandardThreadExecutor | PT-per-request: provides a built-in concurrency boundary and queuing;
  • StandardVirtualThreadExecutor | VT-per-request: requires explicit application-level concurrency control.

Both have comparable performance characteristics in low-concurrency scenarios.

“Useful concurrency”, N can be one of the decisive factors to select between models:

  • For low N, platform worker thread pools usually cope well;
  • For moderate N, both models can work, benchmarks should guide the decision;
  • For high N with blocking I/O workloads, virtual threads become increasingly interesting.

The final choice also depends on the preferred overload-absorption model: sized queuing or timeout-bounded waiting, as well as on whether the critical request path contains operations that may pin virtual threads.

Lastly, StandardVirtualThreadExecutor is relatively new in Tomcat, and VTs are still new in the Java ecosystem in general; therefore, it is worth watching how the implementation, tooling, and engineering practices around them evolve.
