Spring Boot | Tomcat execution models
This article explains how Spring Boot with embedded Tomcat executes blocking HTTP requests using either a classic worker thread pool or virtual threads. It compares scalability, resource usage, queueing behavior, overload modes, and provides a practical heuristic for choosing between them.

Virtual threads in Java are often discussed in isolation, detached from the realities of enterprise application development.

But their practical value only becomes clear when placed inside the code execution model most Java teams actually use: Spring Boot running on a Servlet container such as Apache Tomcat.

This article compares how that same blocking programming model behaves when requests are executed by traditional platform thread pools (PTs) versus newer virtual threads (VTs).

The goal is to build a practical mental model how Spring Boot and Tomcat execute blocking requests, identify differences in scalability, resource usage, failure modes, trade-offs and guide how to choose between both models.

But, first let’s step back and look at the broader evolution of the Spring stack.

Evolution#

The evolution milestones below cover only Spring MVC (servlet-based), whereas Spring WebFlux (reactive path) is out of scope for this article.

Milestone #1: platform thread pools, sync blocking, ~2004–today#

In the classic Spring MVC/Tomcat model, request execution was built around a platform thread pool managed by the Servlet container.

Each incoming HTTP request was assigned to a real OS-backed PT, which executed the Spring stack, controller code, service logic, and any blocking database or downstream calls.

This model worked well and became the industry default; it was simple and productive, but under high-concurrency I/O-heavy workloads, blocked requests consumed scarce worker threads, forcing teams to tune large pools, accept higher memory usage, or reject/queue more requests.

Spring Boot later adopted embedded Tomcat and simplified deployment/configuration (~2014), but the fundamental execution model remained the same.

Milestone #2: servlet async, delegated blocking ~2009/2010–today#

Async Servlet (v3.0, ~2009) processing introduced a way to start a request on a container thread, release that thread while longer work continued elsewhere, and complete the response later.

Spring MVC exposed this model through a dedicated API for long-running requests, delayed responses, streaming, and cases where keeping a Tomcat worker PT blocked for the whole request lifetime was wasteful.

However, it usually did not remove blocking; it relocated it to another PT-based executor and introduced a more asynchronous programming style.

So this model improved request-thread utilization, but it did not remove platform thread pool management and remained a specialized tool rather than replacing the classic synchronous Spring MVC model.

Milestone #3: virtual threads, sync blocking, ~2023–today#

In June 2023, Tomcat added support for VT-based request execution through a dedicated virtual-thread executor, and Spring Boot made this model easy to enable from configuration.

VTs changed the trade-off by making the classic thread-per-request model much cheaper for I/O-heavy workloads, which is why they are often described as a “game changer”.

When the request blocks on a database or downstream service, the VT can park and unmount from a carrier PT, allowing much higher concurrency with the familiar blocking programming model.

However, this also creates a new risk: if the application can accept far more concurrent requests, then explicit app-level concurrency limits, downstream bulkheads, timeouts, and load shedding become mandatory rather than optional.

It is worth noting that Tomcat’s internals have their own rich history of change and refinement.

For the fuller story, refer to Mark Thomas’s presentation.

Brief blocking stack review#

Today in Spring Boot, PT-per-request and VT-per-request coexist as two practical blocking request execution models for enterprise Java workloads: from milestones #1 and #3.

Each has conditions where it fits best, so the rest of this article compares them directly.

How Spring Boot bootstraps Tomcat?#

Prior to diving deeper into the execution models differences, it is worth reviewing how Spring Boot spins up Tomcat and configures it.

Since Spring Boot 3.2, it provides an easy way to switch to VT-based execution via externalized configuration (application.yaml):

spring:
  threads:
    virtual:
      enabled: ${VT_ENABLED:false}

Internally, Spring Boot customizes Tomcat’s request-processing executor.

Spring Boot uses an application-level setting for this case, because this flag affects not only Tomcat, but also other places where Spring manages execution, such as task execution, scheduling, async processing. The reason for this is explained in Brian Clozel’s comment.

Additionally, Spring Boot provides a set of Tomcat-specific settings with the prefix server.tomcat.* meant to be overridden through externalized configuration.

The most relevant settings for this comparison are:

  • accept-count;
  • max-connections;
  • threads.max, PT-specific;
  • threads.max-queue-capacity, PT-specific.

When Spring Boot starts embedded Tomcat programmatically as part of the Servlet web-server startup, it applies those settings through Tomcat customizers and eventually passes the values to Tomcat’s own runtime.

Then Tomcat manages a great share of the full request path, see next section.

How Tomcat executes requests?#

At a high level, Tomcat’s request path can be viewed as 3 logical layers:

  • Connector / Network: enables servlet container to function as a stand-alone web server; lowest-level layer in which Tomcat operates at TCP connection and socket levels cooperating with OS directly through Java NIO API.

  • Coyote / Protocol: Tomcat’s HTTP protocol engine (logically part of Connector). It handles HTTP/1.1 and HTTP/2, converts socket activity into Tomcat’s internal request/response model, and bridges the exchange into Catalina’s servlet pipeline.

  • Catalina / Servlet: maps the request to the web application, applies the Servlet pipeline, invokes filters and eventually reaches Spring MVC’s DispatcherServlet, from which Spring Boot’s own execution begins.

During startup, when Tomcat binds the server socket, it passes acceptCount param (= 100 by default) as the Java NIO socket backlog argument, effectively asking the OS to size the TCP listen backlog for incoming connection attempts not yet accepted by Tomcat.

Inside logical Connector, the work is split (see UML-sequence diagram):

  • Acceptor accepts new socket connections (capped by maxConnections param (= 8192 by default));
  • Poller watches already accepted sockets and detects when they are ready for reading or writing; it then submits SocketProcessor tasks to the configured executor and further processing happens on a worker thread: VT or PT.

Thus, logical Connector is exactly where VT vs PT separation begins. Both params are relevant for both execution models and are explicitly shown on Fig. 1 and Fig. 2 see below.

Request processing model #1: PT-per-request#

Fig. 1 shows a simplified request path inside Apache Tomcat when requests are executed on platform threads from a worker pool.

In this model, a normal blocking request occupies one PT for the whole request-processing path: from Connector/Coyote, through Catalina/Servlet pipeline, into Spring Boot app code.

After socket readiness is detected, the SocketProcessor tasks is submitted to StandardThreadExecutor implementation, which is backed by a worker thread pool and Tomcat’s specially designed TaskQueue, which inverts the priority in thread usage.

When all current workers are busy and the pool can still grow, Tomcat creates additional PTs. After the pool reaches its configured limit (maxThreads param (= 200 by default), new tasks are queued instead.

When load decreases, excess idle workers are not kept forever: Tomcat retains at least minSpareThreads and retires extra idle PTs after the configured idle timeout.

As a matter of fact this queue is virtually unbounded (maxQueueSize param (= Integer.MAX_VALUE by default)) and does not have a configurable timeout that expires queued tasks.

This means that a large number of tasks may accumulate there, leading to growing latency, memory pressure, and slow recovery from overload.

If this queue is bounded and becomes full, submitted work is rejected by default, resulting in forcing closing the TCP connection.

Spring Boot started to expose a direct config parameter for this queue only relatively recently, after issue submitted by Moritz Halbritter (~ 2023).

This highlighted the importance of explicitly bounding the queue based on a particular workload.

Overall, this model is simple, mature, and still a good fit for many Spring Boot enterprise applications.

Its main advantages are:

  • Worker thread pool gives a clear concurrency limit;
  • Queueing lets absorb short overload spikes not rejecting too early;
  • Has proven to work well for mixed or CPU-heavy workloads with moderate concurrency.

Its main limitation comes from the 1:1 mapping between PTs and native OS threads:

  • High I/O concurrency translates directly into more native stack memory, more OS-visible threads, and higher scheduling overhead.

Request processing model #2: VT-per-request#

Fig. 2 shows a simplified request path inside Apache Tomcat when requests are executed on virtual threads.

In this model, a normal blocking request occupies one VT for the whole request-processing path, instead of holding a platform thread forever.

In this case, Tomcat submits SocketProcessor object to StandardVirtualThreadExecutor implementation, which is much simpler than StandardThreadExecutor.

Every ready socket is immediately assigned to a newly created VT, virtually removing natural thread pool concurrency cap and any explicit queueing.

Also, Tomcat does not impose any additional concurrency limit, not configurable semaphore, not anything, putting the responsibility on an application developer.

VTs use small, dynamically growing heap-stored stacks, typically measured in KB and are scheduled by the JVM onto a small number of underlying platform threads (carriers) this is the M:N scheduling, which in turn are mapped 1:1 to OS threads.

As stated in JEP 444, “carrier thread pool” is implemented by a dedicated instance of ForkJoinPool, distinct from the common pool used in parallel streams, and is backed by a small number of platform threads equal to the number of available CPU-cores.

When a VT blocks (e.g., on I/O), it is parked by the JVM, unmounts from a carrier, allowing the carrier thread to run other tasks.

This makes this model extremely adapted to highly-concurrent applications with many blocked in-flight VTs.

At the same time, this introduces an intermediate scheduler that parks and resumes VTs, which at scale may add extra churn for CPU-bound workloads and high-RPS I/O-bound workloads with little blocking.

Its main advantages come from VTs advantages:

  • Cheaper I/O concurrency, meaning less waste of native thread stacks;

Its main limitation (as of Java 25) is:

  • Pinning can reduce scalability when a VT cannot unmount from its carrier (native method or a foreign function);

Hom models behave under pressure?#

To understand the practical difference between the 2 execution models, it is not enough to look only at the request flow.

The real difference becomes visible under load: when concurrency grows, capacity is reached, and the system starts to resist.

For this reason, I will model the behaviour of PT-based and VT-based execution in 3 situations, as shown in Fig. 3:

  • Normal, low concurrency, N < 10;
  • Saturation, high concurrency, capacity is reached;
  • Degradation, extreme concurrency, N >> Nnorm

More explanation of these regions are in this article.

PT-based execution under pressure#

Normal: the system has enough idle worker PTs, stable processing

Tomcat accepts the next TCP connection, waits for the Poller to confirm socket readiness, and submits a SocketProcessor task to the StandardThreadExecutor.

The executor can immediately hand the task to an available worker PT. Further processing runs on that PT along the full request path. On a blocking I/O call, the worker PT remains occupied by that request until the I/O completes.

The system is stable and clients receive responses with low latency.

Saturation: worker PTs become the limiting resource, queueing begins

At this load point, often around N ≈ maxThreads for blocking workloads, all worker PTs inside the StandardThreadExecutor are busy processing other requests.

New SocketProcessor tasks start spending more time in the executor’s TaskQueue until a worker PT becomes available. The larger the gap between the rate of incoming work and the rate of completed requests, the faster the queue grows.

This causes latency to increase super-linearly. Throughput no longer grows proportionally with concurrency and eventually reaches a plateau.

Clients or an intermediary such as a load balancer can still time out.

Degradation: the system moves from slow responses to failed connection attempts

Depending on the HTTP protocol version and connector configuration, the TaskQueue may become full and reject new tasks, causing TCP connections to close.

Alternatively, the Acceptor may reach its maxConnections limit forcing new connection attempts into the OS accept backlog up to the configured acceptCount.

This queue is relatively small, so once it becomes full, the OS may start refusing new TCP connections.

Exact behavior is OS- and config-specific, but the result is the same: new TCP connections are no longer established reliably.

Clients start seeing connection timeouts.

VT-based execution under pressure#

Normal: the system keeps issuing new VTs, stable processing

Tomcat accepts the next TCP connection, waits for the Poller to confirm socket readiness, and submits a SocketProcessor task to the StandardVirtualThreadExecutor.

The executor runs the task on a new VT. Further processing continues on that VT along the full request path. On a blocking I/O operation, JVM may park and unmount VT from carrier and mount back when response is ready.

The system is stable and operates normally with low latency.

Saturation: app-level concurrency limit is reached

At this load point, if an app-level concurrency limit, for example Semaphore-based, is set and reached, the system either rejects requests immediately, usually translating them into HTTP 429 responses, or makes requests wait for a permit, causing app-level queueing and increased latency.

In the absence of app-level concurrency restrictions, excessive request concurrency may instead overwhelm downstream services or cause queuing on connection pools.

Client experience then depends on the downstream behavior: from increased latency and explicit errors to connection-pool exhaustion, for example reaching the maxConnections limit in HTTP/1.1 clients.

Degradation: the increased load is no longer efficiently translated into goodput

Assume an app-level concurrency limit is set. Load beyond the limit produces more rejected requests, usually translated into HTTP 429 responses. This protects the downstream path, but rejection is not free: exception-heavy or log-heavy rejection can still consume CPU and memory.

As the rejected share grows, goodput — successful 2xx responses — drops and the server enters the degradation regime. Some requests still complete successfully, while most excess clients receive immediate 429 responses after reaching the app.

Recovery is usually smoother than with uncontrolled overload because excess work is rejected near the app boundary instead of accumulating deeper in the system.

Let’s talk about “useful concurrency”?#

The load modeling shows that in both cases - PT-based or VT-based execution - concurrency cap is mandatory for any production system with non-trivial load.

The core difference is that PT-based executor implicitly restricts concurrency (N), and VT-based executor relies on a developer to do it explicitly at application level.

Ignoring this boundary does not create “unlimited scalability” but rather leads to unpredictable consequences under pressure, mentioned above.

Secondly, it is essential to figure out what is actually maximal useful concurrency for a particular application.

By useful concurrency, I mean the maximum number of in-flight requests the system can sustain without overwhelming downstream dependencies or the server instance itself.

Once this number is roughly understood, it can guide both execution models:

  • PT, size the Tomcat worker thread pool and task queue;
  • VT, set explicit app-level limits, such as semaphore permits.

Lastly, it is useful to estimate the expected range of useful concurrency to decide which model fits best.

The decision tree below uses exactly this question as the selection criterion.

The decision tree to support choice#

Fig. 3. shows a decision tree to support decision on which model to use.

The tree is derived from JEP 444, which states that VTs can significantly improve throughput when:

  • The workload is not CPU-bound.

  • The number of concurrent tasks is high.

Since enterprise workloads are often mixed, the core question becomes: “How much concurrency is too high for worker thread pools?”

With Tomcat’s PT-based executor, request-processing concurrency is already bounded at N=200 by default, where PTs usually cope well and switching to VTs is rarely necessary.

This limit can be increased; in a typical Linux environment, there is no hard, non-configurable PT-per-process limit.

The practical constraint is memory, mainly the native stack cost of platform threads.

On Linux x64, a native thread stack can reserve up to 1024 KB, while initially committed memory can be ~136 KB and grow lazily on demand.

So the question becomes: “How expensive is it to raise the worker pool size?”

An extra 100 PTs may cost roughly 13.6–100 MB, which is often acceptable; an extra 1000 PTs may cost 136 MB–1 GB, which can already become a serious part of the container memory budget.

That is why I use N=1000 as a rough practical line: not a hard limit, but a point where VTs usually start to look better for memory usage.

Ths makes N=200–1000 range a mixed territory: both models can work, and the decision depends on CPU saturation and memory constraints.

If requests mostly wait on I/O with moderate CPU usage, VTs become more attractive because they let us save memory.

If requests mostly consume CPU with high RPS rate, VTs do not create more CPU capacity and may even behave worse because of additional scheduler overhead.

Having said that, the tree should not be treated as a strict rule, but as a practical heuristic.

And the final decision should still be validated with benchmarks on the target hardware and expected workload.

Conclusion#

Platform worker thread pools are not obsolete, and virtual threads do not provide “unlimited scalability”.

VTs make blocked in-flight requests cheaper, but they do not remove real bottlenecks such as CPU, memory, database pools, connection pools, locks, or downstream services.

Apache Tomcat supports both models through a dedicated Executor: either StandardThreadExecutor (PT-based) or StandardVirtualThreadExecutor (VT-based).

StandardThreadExecutor provides a built-in concurrency boundary, whereas StandardVirtualThreadExecutor requires explicit application-level concurrency control.

Both have comparable performance characteristics in low-concurrency scenarios.

“Useful concurrency”, N is one of the decisive factors to select between models:

  • For low N, platform worker thread pools usually cope well;
  • For moderate N, both models can work, benchmarks should guide the decision;
  • For high N with blocking I/O workloads, virtual threads become increasingly interesting.

The proposed decision tree should be treated as a rough heuristic rather than a strict rule.

The final choice also depends on whether the critical request path contains operations that may pin virtual threads.

Lastly, StandardVirtualThreadExecutor is relatively new in Tomcat, and VTs are still new in the Java ecosystem in general. Therefore, it is worth watching how the implementation, tooling, and engineering practices around them evolve.
