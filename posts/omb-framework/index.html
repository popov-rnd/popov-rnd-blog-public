OpenMessaging benchmark framework

"Review of a benchmarking framework for measuring performance of streaming and messaging systems"

In today’s post I will review the OpenMessaging benchmark framework (OMB, for short) — an open-source project by The Linux Foundation written in Java designed specifically for benchmarking messaging and streaming systems. It is one of the very few tools purpose-built for this class of systems.

While tools like Grafana k6 and Gatling excel at generating stateless request load and measuring client-side latency, they are not suitable for benchmarking streaming and messaging systems because they lack native concepts of coordinated producers and consumers, backlog observation, end-to-end latency tracking, and time-dependent stability — the core properties from which messaging /streaming performance actually emerges.

Individual streaming and messaging systems often provide their own perf-test or CLI tools (e.g. Kafka Rabbit, Artemis). While useful for tuning a single system in isolation, these tools are not designed for cross-system comparison.

For this reason, I decided to take OMB “on board” and explore it in-depth. The official documentation is relatively sparse, so this post provides a deeper interpretation based directly on the open-source code and observed behavior.

High-level#

At a high level, OMB is a multi-module Maven project (GitHub source) that you can build and package manually. The core framework and each supported messaging or streaming system are implemented as separate Maven modules, allowing new systems to be added in a pluggable way.

Each system-specific module implements a common OMB driver API, primarily:

  • BenchmarkDriver - lifecycle management, topology and orchestration;
  • BenchmarkProducer - message publishing logic;
  • BenchmarkConsumer - message consuming logic.

These interfaces define how OMB interacts with a messaging system in a uniform, system-agnostic way.

Under the hood, each driver module uses the official Java client libraries of the target system (e.g. Kafka client, RabbitMQ client, Artemis client) and delegates all message operations directly to the system’s native APIs.

OMB itself does not implement messaging semantics or protocol logic; it only orchestrates execution and collects metrics.

Architecture#

At an architectural level, the OMB framework consists of 3 primary components:

  • Benchmark (coordinator): the benchmark entry point, a JVM process that serves as the control plane. It dispatches execution tasks, and collects per-worker metrics to compute final aggregated results.
  • BenchmarkWorker: the worker entry point, one or more JVM processes acting as execution engines. Workers receive tasks from the coordinator, instantiate producers and consumers using the configured driver, generate message traffic against the target system, and record latency and throughput metrics locally before reporting them back.
  • Target brokers: independently deployed messaging or streaming systems (e.g. Kafka, RabbitMQ, ActiveMQ Artemis). These systems expose their native network endpoints and credentials to the benchmark, but remain otherwise unaware of OMB.

Workers are typically launched on separate machines to avoid shared CPU, memory, and network bottlenecks. Inside an OMB worker, Javalin is used to expose those http endpoints, which is a minimal, code-first HTTP server for infrastructure and tooling, powered by Jetty behind the scenes.

All message traffic flows directly between workers and the target system using native protocols; HTTP is used only for control and metrics exchange between the coordinator and workers.

Getting started#

As I already mentioned, OMB is a Maven-based project that can be compiled and packaged using standard commands:

mvn clean package

Once packaged, a folder package/target/ contains an archive:

openmessaging-benchmark-0.0.1-SNAPSHOT-bin.tar.gz

You just extract the content of this archive in any working dir.

Once extracted, you will see bin/ folder containing several bash scripts:

  • benchmark: bash script that launches a coordinator process;
  • benchmark-worker: bash script that launches an execution engine process.

Benchmark is launched as a JVM coordinator process with two YAML config files passed as startup params:

  • Driver: config. defining the target system (system-specific settings: url address, credentials, etc.);
  • Workload: config. describing the benchmark scenario (system-agnostic params: producers, topics, consumers, rates, duration).

Based on these configs, OMB orchestrates producers and consumers, controls publish rates and payload sizes, and collects metrics such as throughput and latency throughout the run.

Generalized benchmark launch command looks like this:

./bin/benchmark \
   --drivers <driver-path>[,<driver-path>...] \
   [--workers <worker-url>[,<worker-url>...]] \
   <workload-path> [<workload-path>...]

where

  • —drivers defines comma separated list of systems being benchmarked sequentially;
  • —workers [optional] defines a comma separated list of separately deployable workers URLs.
  • workload-paths define a whitespace-separated list of workload files paths.

For full set of supported params, see the entry point class Benchmark.Arguments.

Workers are launched on target machines prior to launching the benchmark itself:

./bin/benchmark-worker \
   [--port <http-port>] \
   [--stats-port <metrics-port>]

where

  • —port [8080 by default] defines http a port number exposed to the coordinator to receive instructions and response with runtime metrics;
  • —stats-port [8081 by default] defines a port number exposed to metrics scrapers (Prometheus);

For full set of supported params, see the corresponding entry point class BenchmarkWorker.Arguments.

Below are several examples of how the benchmark can be launched.

This command launches benchmark process with single local worker, being created at start-up (simplest set-up for quick evaluation).

./bin/benchmark \
   --drivers driver-popov-rnd-rabbit/rabbitmq.yaml \
   workloads-popov-rnd/simple-workload-1kb.yaml

This command launches benchmark process with two remote workers, being pre-created and available via http at given IP addresses (more common for prod. environments).

./bin/benchmark \
   --drivers driver-popov-rnd-rabbit/rabbitmq.yaml \
   --workers 1.2.3.4:8080,4.5.6.7:8080 \
   workloads-popov-rnd/simple-workload-1kb.yaml

In this set-up, each worker independently runs its assigned slice of the same benchmark workload and reports its own metrics, which the coordinator later merges into a single global result.

Workloads#

Benchmark scenarios are described in workload YAML files. Full set of keys being supported is controlled by Workload class. Below is a concise overview of the most commonly used parameters.

  • name: (string) workload name being shown in reports and console;
  • topics: (int) number of topics (front doors: exchanges, addresses, topics, etc.);
  • partitionsPerTopic: (int) for streaming platforms only, self-explained;
  • keyDistributor: (enum) [NO_KEY, KEY_ROUND_ROBIN, RANDOM_NANO] key distribution strategy for partition selection, see class KeyDistributorType;
  • messageSize: (int, bytes) the size of each message in bytes, like (10, 100, 1024, 2048, etc.);
  • useRandomizedPayloads: (boolean) true or false, whether to randomize payload content, see WorkloadGenerator, public run() method;
  • randomBytesRatio: (double) used in conjunction to useRandomizedPayloads = true, the ratio of bytes in each message that are randomized (0.0–1.0);
  • randomizedPayloadPoolSize: (int) used in conjunction to useRandomizedPayloads = true, the number of randomized payloads cached in memory;
  • payloadFile: (string, optional) a path to the file containing payload data, used only as a byte pattern source, must match messageSize if present, pre-defined files are listed in Payload folder;
  • subscriptionsPerTopic: (int) number of individual consumer groups (streaming) or separate queues (messaging) per topic;
  • producersPerTopic: (int) number of independent producer instances publishing to a topic;
  • consumersPerSubscription: (int) number of consumer instances within a single subscription (i.e., a consumer group in streaming systems (streaming), or competing consumers attached to the same queue (messaging);
  • producerRate: (int, messages/sec) producer send rate, 0 (zero) means unlimited (push as fast as possible, iteratively increased until max is reached, see WorkloadGenerator class, its private method findMaximumSustainableRate(double currentRate)));
  • consumerBacklogSizeGB: (long, defaults to 0) size of backlog (in GB) to accumulate before consumers start, 0 (zero) means consumers start immediately;
  • backlogDrainRatio: (double, defaults to 1.0) fraction of backlog that must be drained for test to complete; 1.0 means full drain; 0.0 means do not wait for drain;
  • testDurationMinutes: (int, minutes) the duration of measurement phase (minutes), typically [5-15];
  • warmupDurationMinutes: (int, minutes) the warmup phase duration (minutes, not counted in results), typically [1-2]

Most important keys (for a start) are highlighted in bold. OMB also provides a set of predefined workloads in the workloads/ directory, which can be used as a starting point for experiments.

Results#

OMB computes metrics in fixed 10-second reporting windows. At the end of a benchmark run, these windowed statistics are aggregated and emitted as a JSON (by default) result file, stored in the directory from which the test was launched. For how this file data is formed, see class WorkloadGenerator, its private method printAndCollectStats(). For exact fields, see TestResult class.

OMB does not provide an official or stable schema for this result JSON (as of the moment of writing this post). Therefore, this section serves as the authoritative interpretation layer for the metrics referenced in the JSON.

Below are key metrics (not all(!)) that the resulting JSON report contains. Metrics logically fall into 3 groups: throughput (rates, backlog), broker latency (publish latency), and user-visible latency (end-to-end latency).

  • publishRate: (List<Double>, messages/sec) the effective message publish rate, measured as messages per second over fixed 10-second windows; tells you what rate the system actually sustained. When producerRate is set to 0 (unbounded) in a workload, the values represent the maximum publish rate OMB was able to achieve under the given workload and system constraints. When producerRate is explicitly defined, this series largely mirrors the configured rate, assuming the target system can sustain it.
  • consumeRate: (List<Double>, messages/sec) the effective message consume rate, measured as messages per second over fixed 10-second windows; this metric indicates how fast consumers are able to drain messages from the system. In a stable workload, consumeRate closely matches publishRate; sustained divergence between the two indicates backlog accumulation, consumer-side bottlenecks, or broker-side delivery constraints.
  • backlog: (List<Long>, messages) the number of messages produced but not yet consumed, sampled over fixed 10-second windows. A backlog close to 0 (zero) indicates a stable system where consumption keeps pace with production; a steadily growing backlog signals overload or delivery bottlenecks, regardless of the reported publishing throughput.
  • publishErrorRate: (List<Double>, errors/sec) the rate of publish failures sampled over fixed 10-second windows. Any sustained non-zero value indicates broker-side saturation or client-side send failures and invalidates throughput and latency measurements for that interval.
  • publishLatency_: a set of time-series metrics (List<Double>, milliseconds) representing the time from producer send to broker acknowledgment. This latency includes protocol handling, persistence / fsync, replication (if enabled), and broker scheduling. Start by examining the publishLatency95pct and publishLatency99pct arrays to identify when (in which 10-second window) the broker begins to struggle; spikes in these tail latencies typically precede backlog growth and eventual throughput collapse.
  • aggregatedPublishLatency_: a set of aggregated latency metrics (double, milliseconds) summarizing the entire benchmark run. These values are computed from a cumulative latency histogram that spans the full run and are not derived from the per-window arrays. They describe the broker’s overall latency envelope under the given workload, independent of when individual spikes occurred, and are best suited for cross-run and cross-system comparison.
  • publishDelayLatency_: a set of time-series metrics (List<Long>, microseconds) is not a performance metric but rather a control metric that shows whether the benchmark configuration, rather than the broker, is limiting throughput. It shows how long the producer waits because a producerRate is configured. Higher values mean the benchmark is waiting on purpose to keep the send rate at the configured limit, not because the broker is slow.
  • aggregatedPublishDelayLatency_: a set of aggregated publish latency metrics (long, microseconds) summarizing the entire benchmark run. These values are computed from a cumulative latency histogram spanning the full run and are best used for cross-run and cross-system comparison, not for diagnosing when latency spikes occurred.
  • endToEndLatency_: a set of time-series metrics (List<Double>, milliseconds) measuring the time from message publish to message consumption. This includes broker queueing, persistence, scheduling, and consumer lag. OMB computes this latency as the difference between the timestamp recorded at message send and the timestamp recorded at message consumption. Start by examining endToEndLatency95pct and endToEndLatency99pct: these values represent the slowest 5% and 1% of messages, respectively, and therefore define the worst delays that downstream consumers observe.
  • aggregatedEndToEndLatency_: a set of aggregated end-to-end latency metrics (long, milliseconds) summarizing the entire benchmark run. These values are computed from a cumulative end-to-end latency histogram spanning the full run and represent overall message delivery behavior rather than moment-to-moment variation. This is the primary latency signal for latency-focused workloads and is best used for SLA-style evaluation and cross-run or cross-system comparison.

Most important keys (for a start) are highlighted in bold. Focus on these first to assess system stability and capacity before diving into secondary metrics.

Limitations I faced#

OMB is primarily streaming-oriented, and its terminology — topics, subscriptions, partitions — reflects that focus, which can be confusing when evaluating classic message brokers. Some its drivers (as Apache Artemis) don’t natively support shared-queue contention scenarios, such as multiple producers and consumers on a single queue. As a result, some messaging semantics require careful interpretation or complementary testing tools.

While driver YAML files expose many tuning options, some crucial behaviors remain hard-coded inside the driver implementation and cannot be adjusted without changing the code (e.g. publish confirm is enabled by default and cannot be easily switched to fire-and-forget mode).

Regarding latency precision in multi-worker setups, a project maintainer notes that end-to-end latency is computed using timestamps taken on different nodes, which makes it inherently less reliable, see issues. In practice, this can be mitigated by synchronizing clocks using Network Time Protocol (NTP); e.g. on AWS EC2, this is provided out of the box via the Amazon Time Sync service.

OMB is developed, tested, and primarily operated on Linux. Most documentation, scripts, and implicit assumptions (filesystem layout, networking, tooling) target Linux environments, so non-Linux setups tend to introduce additional friction rather than being first-class, even though Docker image is supported.

Be aware, the project uses Lombok throughout the code base, so a Java version upgrade should be paired with a corresponding Lombok version upgrade. Why Lombok may break the compilation I explained in this post.

Overall, despite these limitations, OMB remains unique in providing a vendor-neutral, reproducible, and code-inspectable benchmark designed around streaming semantics, which can be adapted, sometimes imperfectly, to evaluate classic messaging systems.
