JVM vs native: performance analysis

JVM vs native: performance analysis
This article compares JVM and GraalVM native executables across major Java frameworks, focusing on throughput, latency, and resource usage under load.
jvm, graalvm, performance, benchmark

The performance gap between the same Java application running on the JVM and compiled with GraalVM into a native executable is often assumed without actually realizing the magnitude of the difference.

Most performance comparisons today focus on memory footprint and startup time, while fewer analyses address runtime metrics such as throughput and latency.

There are relatively few benchmarks that compare JVM and native modes for major Java frameworks (Spring Boot, Quarkus, Micronaut) in a reproducible and unbiased way. A recent study by the Quarkus team is one of the few public references.

In this article, I aim to address that gap by conducting my independent evaluation.

I looked at the systems from 3 angles:

  • Throughput scaling (closed workload model);
  • Latency under load (open workload model);
  • Resource usage (CPU and memory under load).

In this article, “JVM mode” refers to running the application on the JVM, while “native mode” refers to an ahead-of-time (AOT) compiled native executable (GraalVM Native Image).

Objectives#

The goal was to build an equivalent minimal application in each framework and evaluate their behavior under identical workloads and configuration constraints.

Specifically:

  • Compare throughput between JVM and native modes;
  • Evaluate whether this difference is consistent across Spring Boot, Quarkus, and Micronaut;
  • Compare resource usage (CPU and memory) between modes.

The focus is on relative differences rather than absolute values. Cross-framework comparisons are not the primary focus of this study.

Workload design#

Given my objectives, I considered the following designs:

  • GET /plaintext HTTP/1.1;
  • GET /json HTTP/1.1;

However, I chose a different setup:

  • POST /ingest HTTP/1.1 with JSON body ~300 Bytes; a set of payloads was pre-generated in advance.

The goal was to model a real-world scenario, where JSON parsing and validations (via Jakarta Validation) are part of the critical path.

Each request is routed to a controller, parsed into a Java object, validated at the field level, and ends with 204 No Content response.

Experiment set-up#

To ensure objectivity, I created 3 separate Java projects based on Spring Boot, Quarkus, and Micronaut (source code will be provided at the end).

Application bootstrap#

Each project uses the minimal dependencies required to expose the web layer (HTTP server, JSON binding, validation), following each framework’s recommended setup.

The goal was to compare realistic “minimal viable” applications — not an artificially normalized dependency set.

The underlying web server was not unified; each framework was used with its default stack: Spring Boot uses Tomcat (servlet-based stack); Quarkus and Micronaut uses Netty (event-loop).

All applications expose an equivalent minimal endpoint: POST /ingest without business logic, returning 204 No Content.

All applications were built and containerized using framework-provided building tools (Maven/Gradle plugins), which automatically produce Docker images in a local registry for both JVM and native builds.

No framework-specific tuning was applied (e.g. GC, thread pools, or server settings). GraalVM native images were also built with default settings (Serial GC, no optimization flags, or target architecture tuning).

An important constraint: execution model could not be fully normalized across frameworks without losing ergonomics. In particular, Micronaut didn’t provide a built-in mechanism to offload request body parsing and validation from the event loop to a worker pool, see issue. As a result, defaults were applied: for Quarkus - executing on a fixed worker pool (200 threads); for Micronaut - executing on event-loop threads.

Execution environment#

Hardware & environment

All benchmarks were executed locally on a dedicated desktop machine:

  • OS: Linux Mint (kernel 5.15.0);
  • CPU: Intel Core i7;
  • Memory: 32 GB RAM;
  • Storage: 500 GB SSD.

Runtime & frameworks

  • Spring Boot v4.0;
  • Quarkus v3.30;
  • Micronaut v4.6;
  • JDK & GraalVM CE JDK v25.

Helping toolset

  • Load generator: Grafana k6 v1.6;
  • Container runtime: Docker Engine v29.1.3;
  • Automation: Bash scripts;
  • Visualization: Google Sheets.

Execution constraints

Each application was containerized and run under identical limits: 2 CPU cores and 1GB RAM:

docker run -d --name $FRAMEWORK-$MODE --cpuset-cpus="0,1" --memory=1G -p 8080:8080 $IMAGE_NAME

Measurement methodology#

2 workload models were used, serving different purposes:

  • Closed workload model (fixed concurrency):

    • used to determine the maximum sustainable throughput (Xmax) for each setup;
    • each concurrency level was executed for 2.5 minutes;
    • average latency and actual achieved throughput were recorded.

  • Open workload model (fixed arrival rate):

    • used to measure latency behavior under controlled offered load;
    • each load level was sustained for 15 minutes;
    • percentiles (p50, p95, p99) and actual achieved request rates were recorded.

For JVM-based setups, an additional 2.5-minute warm-up phase (100 VUs) was applied prior to measurement to ensure JIT compilation of hot execution paths.

Each experiment was repeated 3 times; median values are reported.

Results#

Before looking at the results, here are a few terms used throughout the article.

  • Scalability curves: throughput vs concurrency curves (closed workload model), see the article
  • “Knee”: the point where increasing concurrency no longer yields proportional throughput gains, and latency starts to grow rapidly;
  • Saturation: the region where the system reaches its maximum sustainable throughput and additional load primarily increases latency;
  • Throughput (X): achieved request rate (requests/sec, RPS for short);
  • Concurrency (N): number of simultaneous in-flight requests (virtual users, VUs).

Closed workload model#

In the closed workload model (fixed number of concurrent users), I first explored scalability curves over a wide range of concurrency levels.

I used powers of 2 for concurrency: 1, 2, 4, 8, 16 … up to 1024. This helped to find where the knee starts.

After that, I ran more tests in that region with step = 2, for example: 4, 6, 8, 10 … up to 16 and 32. These points are shown on the charts.

The k6 load generator script is available on GitHub

To compare JVM and native modes:

Maximum throughput (Xmax) was taken as the highest stable request rate under increasing concurrency, representing the system’s capacity limit. The ratio Xmax(JVM) / Xmax(native) was used for comparison.

Spring Boot

Spring Boot curves were a bit tricky. The “knee” appeared around N ≈ 8-12. After that, throughput continued to grow and reached ≈26–27k RPS at N ≈ 256–512, but this was clearly due to request queuing, not real processing capacity.

Because of that, only the region near the knee (Fig. 1.1) is considered meaningful. It clearly shows saturation for both JVM and native setups.

For consistency across frameworks, Xmax is taken at the knee (effectively Xknee).

Peak throughput (Xmax):

  • JVM ~22k RPS vs Native ~12k RPS (≈1.8× higher for JVM)

Quarkus

Quarkus curves’ (Fig. 1.2) saturation appeared much later for both set-ups: JVM- and native-based. Unlike Spring Boot, Quarkus already fully saturated at nearly N=32, so Xmax ~ Xknee in this case.

Peak throughput (Xmax):

  • JVM ~28k RPS vs Native ~17k RPS (≈1.75× higher for JVM)

Micronaut

Micronaut curves’ plateau (Fig. 1.3) for JVM-based app appeared closer to Quarkus but for native-based app in between Spring Boot and Quarkus.

Peak throughput (Xmax):

  • JVM ~27k RPS vs Native ~15k RPS (≈1.8× higher for JVM)

Across all closed workload model experiments, JVM-based setups consistently showed higher throughput than native ones.

This difference becomes most visible around the “knee” region, where native setups reach saturation earlier, while JVM setups continue to scale further.

Open workload model#

In the open workload model (fixed request arrival rate), I measured latency under increasing load.

This model better reflects real-world conditions and exposes saturation effects more clearly, avoiding what is known as “coordinated omission” effect.

Load was increased stepwise (starting from ~0.4–0.6×Xmax, depending on the framework, up to ~1×Xmax) to capture the transition from stable operation to saturation and latency explosion.

Both p95 and p99 latency were recorded; only p99 is shown, as it provided the most stable and comparable results across runs.

Dropped iterations were also observed at higher load levels for all setups. This occurs when the load generator cannot sustain the configured arrival rate, indicating saturation and excessive response times.

The k6 load generator script is available on GitHub:

To compare JVM and native modes:

  • Define a meaningful SLA (e.g., p99 latency < 50 ms or < 30 ms);
  • Identify the maximum offered load for each mode that satisfies the SLA;
  • Compare throughput by taking the ratio of these stable load levels.

Spring Boot

Fig. 2.1 shows p99 latency as a function of offered load under the open workload model for Spring Boot. Latency remains stable at moderate load levels but increases sharply as the system approaches saturation, indicating the onset of queueing.

Dropped iterations were observed at X=14k RPS for native setup and X=26k RPS for JVM setup, confirming that the generator could no longer sustain the configured arrival rate.

Throughput difference under SLA (p99) constraints:

  • < 50 ms: JVM ~22k RPS vs Native ~12k (≈1.8× for JVM);
  • < 30 ms: JVM ~22k RPS vs Native ~10k (≈2.2× for JVM).

Quarkus

Fig. 2.2 illustrates p99 latency under increasing offered load for Quarkus. The system operates in a stable regime up to saturation, after which queueing dominates and latency grows rapidly.

Dropped iterations and latencies spikes appear at ~12k RPS (native) and ~20k RPS (JVM), marking the limits of sustainable throughput.

Throughput difference under SLA (p99) constraints:

  • < 50 ms: JVM ~20k RPS vs Native ~10k (≈2.0× for JVM);
  • < 30 ms: JVM ~18k RPS vs Native ~10k (≈1.8× for JVM).

Micronaut

Fig. 2.3 illustrates how p99 latency evolves with increasing offered load for Micronaut. The system operates in a stable regime up to saturation, after which latency grows rapidly due to queueing effects. Dropped iterations at ~14k RPS (native) and ~24k RPS (JVM) mark the point where the configured load exceeds sustainable capacity.

Throughput difference under SLA (p99) constraints:

  • < 50 ms/ < 30 ms: JVM ~20k RPS vs Native ~12k (≈1.65× for JVM).

As in the previous experiment, JVM-based setups consistently achieved higher throughput than native ones across all open-loop scenarios.

At the same offered load, JVM mode consistently shows ~1.5-2.5× lower p99 latency compared to native.

Resource usage analysis#

The picture would be incomplete without analyzing CPU and memory consumption under load.

Unlike many benchmarks that report resource usage at startup or idle, the focus here is resource behavior at meaningful operating points — when the system is actually doing work.

The key questions were:

  • At which load level should resource usage be measured?
  • How should CPU and memory consumption be measured reliably?

To answer the 1st question, I reused the results from the open workload experiments and selected the last stable operating points that meet a 50 ms p99 latency SLA. These points represent a comparable and realistic load zone, where both JVM and native setups are still operating without degradation.

To answer the 2nd question, since all applications run in containers, I used Docker’s native stats command

docker stats --no-stream --format "{{.CPUPerc}},{{.MemUsage}}" $CONTAINER_NAME

During each test run (15 minutes), the target container was polled at a fixed interval (1 second), and the metrics were collected into a .csv file.

After the test, the data was analyzed in Google Sheets using standard aggregations such as AVERAGE and MAX. These aggregated values are reported and visualized in the charts below.

Notes on interpretation:

  • CPU percentages above 100% reflect multicore usage and must be interpreted accordingly;
  • Memory metrics include runtime/system overhead (e.g., JVM internals, native allocations);
  • Short spikes may be smoothed due to 1-second sampling.

Spring Boot

Spring Boot application was tested under: 12k RPS in native and 22k RPS in JVM mode in open workload model. The aggregated results are shown on Fig. 3.1.

Resource usage at stable operating points:

  • Memory: Native ~114 MB vs JVM ~519 MB (≈4.5× lower for native);
  • CPU: Native ~177% (~88.5%/core) vs JVM ~149% (~74.5%/core) (higher for native).

Quarkus

Quarkus application was tested under: 10k RPS in native and 20k RPS in JVM mode in open workload model. The aggregated results are shown on Fig. 3.2.

Resource usage at stable operating points:

  • Memory: Native ~36 MB vs JVM ~557 MB (≈15.5× lower for native);
  • CPU: Native ~113% (~56.5%/core) vs JVM ~148% (~74%/core) (higher for JVM).

Micronaut

Micronaut application was tested under: 12k RPS in native and 20k RPS in JVM mode in open workload model. The aggregated results are shown on Fig. 3.3.

Resource usage at stable operating points:

  • Memory: Native ~52 MB vs JVM ~255 MB (≈4.9× lower for native);
  • CPU: Native ~170% (~85%/core) vs JVM ~148% (~74%/core) (higher for native).

Overall, the trend is clear: native containers consistently consume significantly less memory, with reductions ranging from ~4.5× to 15× in these experiments.

CPU usage, however, does not exhibit a uniform pattern. Depending on the framework, native setups may either consume more or fewer CPU cycles.

Conclusion#

This study shows a clear and repeatable pattern.

For the tested request path, across all frameworks and execution models, JVM-mode consistently delivered ~1.4–2× higher throughput, while native mode achieved ~4.5–15× lower memory footprint under high load (near saturation).

CPU behavior was mixed and should be treated as framework-specific rather than as a property of JVM or native alone.

The practical takeaway is that JVM and native optimize for different goals:

  • JVM optimizes performance per instance: more capacity, fewer instances, and lower latency under moderate load.
  • Native optimizes memory efficiency: smaller footprint, more instances per node, and cheaper horizontal scaling.

For highly loaded services, JVM should be treated as the default unless memory is the real limiting factor. Native is not a leaner JVM alternative; it is a different trade-off point with a very strong memory profile and a weaker throughput ceiling.

This result represents a specific slice of the problem space — a synchronous HTTP request path with JSON parsing and validation under constrained resources.

Different workloads, execution models, or tuning strategies may shift these trade-offs, and should be evaluated separately.
