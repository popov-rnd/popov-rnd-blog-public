Virtual threads: performance analysis

"This study evaluates the performance of virtual threads compared to traditional platform thread pools in Spring Boot application, focusing on throughput, latency, and resource usage under IO-bound workload."

Virtual threads: performance analysis
This study evaluates the performance of virtual threads compared to traditional platform thread pools in Spring Boot application, focusing on throughput, latency, and resource usage under IO-bound workload.

The introduction of virtual threads (VT) in Java has generated significant interest, yet their effectiveness in realistic enterprise workloads remains insufficiently explored.

Most public comparisons focus on synthetic scenarios (e.g. thread-sleep-for-n-ms), where VTs naturally scale better due to minimal contention, providing limited insight into real IO-bound systems.

At the same time, practitioners such as Gary DeVal (IBM), Mark Thomas (Tomcat) have reported that under moderate concurrency, VTs often show little advantage over platform thread pools, and may even perform worse.

Additionally, although GraalVM has supported VTs since v22.3 (~2023), there is little to no public analysis of their performance in native mode.

This article addresses that gap through controlled, independent experiments focused on runtime behavior under realistic load conditions.

Objectives#

The goal is to evaluate the practical performance impact of VTs in Spring Boot applications.

The central question is:

“Do VTs provide a meaningful performance advantage over a well-tuned PT config in IO-bound scenario?”

To answer this question, we evaluate the same Spring Boot application under:

2 execution models:

  • Platform threads (PTs): spring.threads.virtual.enabled=false
  • Virtual threads (VTs): spring.threads.virtual.enabled=true

and 2 runtimes:

  • JVM mode (JIT): ./mvnw spring-boot:build-image
  • Native mode (AOT): ./mvnw -Pnative spring-boot:build-image

Each configuration runs the same Spring Boot app, which internally calls a controlled HTTP stub simulating a slow external dependency.

The comparison focuses on: peak throughput, goodput, tail latency and resource usage under load.

About VTs#

The theory behind VTs is not the primary objective of this article.

Still, it is useful to understand how Spring Boot executes app code in both modes, because this helps interpret the results of the study.

By default, Spring Boot uses embedded Apache Tomcat application server, that in turn supports both modes: traditional PT-based and newer VT-based (since v10.1).

Mode #1: platform threads#

Spring Boot defaults running on Tomcat using the platform-thread-per-request model. Tomcat may accept many connections (e.g., maxConnections = 8192), but request execution is limited by the worker thread pool, typically maxThreads = 200; both values are configurable. Each request is handled by a dedicated platform thread (1:1 OS thread), flows through the servlet pipeline, and reaches the controller on that same thread. When all threads are busy, requests queue or may be rejected.

This model is relatively heavy due to OS-level scheduling and per-thread native stack memory, ~1Mb.

Mode #2: virtual threads#

With VTs enabled, Tomcat keeps the same connection handling but replaces the fixed worker pool with a virtual-thread-per-request model. Each request is executed on its own VT, removing the ~200-thread concurrency cap while preserving the same request flow. VTs use small, dynamically growing heap-stored stacks, typically measured in KB and are scheduled by the JVM onto a small number of underlying platform threads (carriers), rather than mapping 1:1 to OS threads. When a VT blocks (e.g., on I/O), it is parked by the JVM, allowing the carrier thread to run other tasks.

This makes the model significantly lighter and better suited for high-concurrency, blocking workloads.

For more elaborate explanation, please refer to the official UML diagrams or this Alibaba blog.

Experiment scheme#

The full experiment scheme is described in Fig. 1. The main application exposes endpoint:

  • GET /io HTTP/1.1 without request body, returning 200 OK response;

Inside the controller, it calls a stub with strictly defined service time (S) and concurrency level (N):

  • GET /delay HTTP/1.1 without request body, returning 200 OK response;

Each request is routed to a controller, inside controller app performs HTTP call to the stub, blocks waiting for response, and ends with 200 OK.

This scenario reflects typical synchronous microservice communication, making it suitable for evaluating threading models under I/O pressure.

All tested versions of the same app (VT/JVM, VT/native and PT/JVM, PT/native) were built and containerized using Spring Boot Maven plugin, which automatically produces Docker images in a local registry.

Applications bootstrap#

Although Spring Boot makes switching to VTs look like a single configuration flag, the change is significant enough that each setup needs to be specified explicitly.

Concurrency consideration#

The VT advantage should appear only under high-concurrency IO-bound workloads, where many requests wait on external I/O and can be parked by the JVM instead of occupying platform threads.

After considering OS-level limitations for a local experiment, I found that a “safe concurrency level” is within ~10k-20k to stay within limits.

HTTP stub was implemented in Go, which provided low memory, high throughput balance compared to Java-based impl-s, with configurable max concurrency and wait time. The code is available on GitHub.

Load regime consideration#

Multiple exploratory load tests were run to understand the impact of key variables: maximum concurrency, service time, HTTP client behavior, CPU capacity, heap size, and container memory.

After this calibration phase, the experiments focused on the following regime:

  • Max. concurrency (N) = 10k
  • Service time (S) = 5 sec

By Little’s Law, this gives an upper bound of throughput: X ≈ N / S = 10,000 / 5s ≈ 2,000 RPS.

This regime keeps CPU utilization moderate while request time is dominated by IO-waiting.

Http client consideration#

This study uses Spring Boot’s Interface Clients: the modern declarative model where Java interfaces define remote endpoints and Spring generates runtime implementations.

The model is built on top of RestClient, the newer synchronous API and by default in Spring Boot uses the JDK built-in http client.

Native JDK client doesn’t expose a traditional configurable connection pool, instead connection reuse and internal threading are managed internally.

Although this may not be the most performant HTTP client setup, it represents mainstream Spring Boot development and was therefore used consistently across all scenarios.

Final application configurations#

  • VT-based app: maxConnections = 20k, maxThreads = 1 (NoOp);
  • PT-based app: maxConnections = 20k, maxThreads = 10k;

Tomcat’s maxConnections was set above the target concurrency level with additional headroom.

In the PT-based setup, Tomcat’s maxThreads effectively acts as the concurrency cap. In the VT-based setup, however, concurrency must be limited explicitly at the app level (e.g. by using Spring Boot’s resilience annotation).

Limiting concurrency at app-level protects downstream services from overload and prevents uncontrolled VT-associated allocation under high load, reducing the risk of OOM errors.

Execution environment#

All benchmarks were executed locally on a dedicated desktop machine:

  • Hardware: Intel Core i7, 32 GB RAM, 500 GB SSD.
  • OS: Linux Mint, kernel 5.15.0
  • Spring Boot 4.0
  • JDK & GraalVM Liberica 25
  • Load generator: Grafana k6 1.6
  • Container runtime: Docker Engine 29.1.3

The stub and all main apps (VT- & PT-based) were run under identical container limits:

  • Stub: 2 CPU cores & 2GB RAM;
docker run -d --name stub --network host --cpuset-cpus="2,3" --memory=2G stub:go
  • Main: 2 CPU cores & 4GB RAM (for JVM containers, heap explicitly restricted at 2Gb, leaving room for platform thread stacks).
docker run -d --name spring-benchmark --network host --cpuset-cpus="0,1" --memory=4G -e VT_ENABLED=$VT_ENABLED $IMAGE_NAME

Additional environment variables were passed to Docker containers were relevant.

Measurement methodology#

The methodology was adjusted for this experiment, using 2 workload models.

  • Step #1: Closed workload model (fixed concurrency); k6 script

    • Objective: measure max throughput, Xmax;
    • Range : from N = 1k to 13k, step 1k;
    • Duration per point: 2.5 min;

  • Step #2: Open workload model (fixed arrival rate); k6 script

    • Objective: measure max “goodput” under steady load;
    • Range : from ~0.5Xmax to ~Xmax with meaningful step;
    • Duration per point: 5 min;

  • Step #3: Resource usage.

    • Objective: compare RAM and CPU usage;
    • Mechanism: native Docker stats CLI & Spring Boot Actuator API;
    • Exercise at selected load point: 15 min, Docker polled each 1 sec;

Additional notes:

  • Note #1: For step #2, the client-side SLA boundary was set to 7 sec. Responses exceeding this boundary were treated as failed from the client perspective.

  • Note #2: For all JVM-based runs, an additional warm-up phase (100k invocations in closed loop) was applied prior to measurement to ensure JIT compilation of hot execution paths.

  • Note #3: For experiments #1 and #2, measurements were repeated 3 times and median values are reported.

Results#

Let me define a few terms used below:

  • Throughput (X, req/sec): the number of completed requests per unit of time.
  • Goodput (G, req/sec): like throughput, but counting only “200 OK” responses.
  • Service capacity (Xmax, req/sec): maximal goodput while system keeps stable.
  • Service time (S, ms): average time a request spends inside the service.
  • Concurrency (N): the number of in-flight requests, in load generators usually approximated by VU (Virtual Users).

JVM-mode#

Closed workload model

First, both apps were explored under a closed workload model across the target concurrency range.

Fig. 2.1 shows that both apps scale almost linearly up to N = 10k.

Across this range, both setups deliver effectively the same successful throughput with peak at N = 10k:

  • Xmax(VT) = 1908 RPS
  • Xmax(PT) = 1894 RPS

The difference is small enough to be treated as measurement variance rather than a meaningful performance gap.

Therefore, in the normal operating range, both setups look almost identical from a throughput perspective.

Open workload model

Secondly, both systems were exercised under an open workload model, where offered load was increased up to the observed capacity limit and then beyond it.

Fig. 2.2 shows goodput as a function of offered load. The chart focuses on the near-knee and overload region, from 1500 to 2300 RPS, where the behavioral difference becomes visible.

Core results from this chart:

  • Stable capacity for both setups: ~1800 RPS
  • p99 latency at stable capacity: VT ~5.7 sec; PT ~5.8 sec

Overload behaviour:

  • VT-based app starts rejecting excess requests with 429, so degradation is explicit and controlled;
  • PT-based app absorbs overload through queueing, which increases latency.

As in the closed workload model, the normal operating range (up to 1800 RPS) does not show a meaningful latency advantage for either setup.

Resource usage

Lastly, both systems were analyzed for resource usage under a fixed offered load of 1800 RPS — the highest load where both systems remained stable and converted almost all offered load into goodput (~99–100 %) within the 7-second client-side SLA.

The container limits were kept unchanged for both runs: 2 CPU cores and 4 GB RAM.

Fig. 2.3 shows Docker container RAM usage during 15-min test. PT-based app uses on average ~1.5x more RAM than VT-based app.

During the same run, both apps used roughly similar JVM heap memory: from ~1.7 GB on average to ~2.0 GB at peak, out of the configured 2 GB maximum heap.

Therefore, the container-level RAM difference did not come from Java heap usage. The most plausible explanation is additional native memory used by platform-thread stacks in the PT-based app.

Additional metrics were also similar, except for process/thread count:

  • CPU usage, AVG: both VT and PT ~78 % (36.5 % per core)
  • Running processes, AVG: VT ~ 500, PT ~ 10700

Native mode#

Closed workload model

As in the JVM-mode experiment, both systems were first evaluated under a closed workload model across the target concurrency range.

Fig. 3.1 shows that both apps repeat the same pattern observed in JVM mode — scale nearly linearly up to N = 10k, but at a lower achieved throughput level.

Across this range, both setups deliver similar successful throughput, peaking at N = 10k:

  • Xmax(VT) = 1616 RPS
  • Xmax(PT) = 1597 RPS

Given the small delta, the result should be interpreted as parity rather than a meaningful performance difference.

In native mode, the achieved goodput was low enough that the configured 10k app-level concurrency limiter was not strongly activated. As a result, the VT-based app produced far fewer 429 responses than in JVM mode: 15–20 % vs 90 %.

Open workload model

Secondly, both systems were exercised under an open workload model, where offered load was increased up to the observed capacity limit (~1600 RPS).

Fig. 3.2 shows goodput as a function of offered load. The chart focuses on the near-knee and overload region, from 800 to 1600 RPS, where the behavioral difference becomes visible.

Core results from this chart:

  • Stable capacity for VT: ~1300 RPS; for PT: ~1100 RPS
  • p99 latency at stable capacity: VT ~6.7 sec; PT ~6.7 sec

Both VT and PT-based app in native demonstrated higher latency, and degraded much faster than in JVM mode.

In this mode, VT-based app was able to sustain up to 15% more load than PT-based, taking into account 7-seconds SLA timeout.

Resource usage

For native mode, resource usage was analyzed under a much lower offered load — 1100 RPS, compared with the JVM-mode experiment — because this was the highest level where both systems remained stable and converted almost all offered load into goodput (~99–100%) within the 7-second client-side SLA.

The container limits were also kept the same for both runs: 2 CPU cores and 4 GB RAM.

Fig. 3.3 shows Docker container RAM usage during 15-min test. Unlike JVM mode, the RAM difference was much smaller: the PT-based app used ~15% more RAM on average and ~12% more at peak.

During the same run, heap usage dominated container RAM usage in both apps: around ~1.5 GB on average and up to ~2.0 GB at peak.

Additional metrics were also similar, except for process/thread count:

  • CPU usage, AVG: VT ~ 53% (26.5 % per core) and PT ~60% (30 % per core)
  • Running processes, AVG: VT ~ 440, PT ~ 8300

The main observation is that thousands of additional PTs did not produce a gigabyte-scale RAM increase. PT used only ~300–400 MB more container memory than VT, which suggests that most native thread stacks used only a small amount of real physical memory during this workload.

Conclusion#

This study did not show a meaningful throughput difference between VT- and PT-based Spring Boot applications in the explored scenario with up to 10k in-flight requests.

In the stable region, both setups performed on par, without visible JVM- or OS-scheduler collapse.

The main difference appeared in the overload region, where request admission strategy became the dominant factor.

  • In the VT-based app, overload was translated into immediate 429 responses;
  • In the PT-based app, overload was absorbed through buffering/queueing at the socket level, leading to higher latency.

Heap usage was broadly comparable between the 2 setups. However, Docker reported higher total memory usage for the PT-based app, most likely due to additional native memory consumed by platform-thread stacks:

  • ~ 1.5× more in JVM mode
  • ~ 15% more in native mode

The experiment also confirmed the significant throughput gap between JVM and native modes. JVM mode delivered around 1.6× higher throughput, consistent with our previous CPU-bound benchmark.

The tested configuration with a 10k Tomcat platform-thread worker pool should not be treated as a canonical setup. It is already in the territory where virtual threads are expected to be a more natural fit.

At the same time, the results show that a properly sized platform-thread pool remains a valid option for moderate-concurrency workloads.

Technical note: why 4 GB was enough for 10k PTs

At first glance, it looks impossible and contradicts the math.

The JVM docs state that platform-thread stack size is controlled by -Xss. On Linux/x64, the default is 1024 KB, which suggests ~10 GB of native memory for 10k PT stacks alone.

However, the configured stack size is mostly a reserved virtual address range. Physical memory is committed gradually, page by page, only as the thread touches the stack — see Volker Simonis.

A blocked thread does not consume the full 1 MB stack; it commits only the pages it actually uses — sometimes as little as ~136 KB.

So the result is not a contradiction: many PTs were created, but not under the worst-case condition where every thread consumes its full reserved stack.
