Java compilation strategies: JIT vs AOT

"A structured look at Java’s compilation pipeline: bytecode compilation, JIT, AOT, and what they imply for performance and deployment."

Java compilation strategies: JIT vs AOT
A structured look at Java’s compilation pipeline: bytecode compilation, JIT, AOT, and what they imply for performance and deployment.
jvm, performance

There is no shortage of articles, blog posts and conf materials on Java compilation, yet after reading many of them, the understanding often remains fragmented.

Individual compilation mechanisms are explained in isolation, often partially or completely detached from official documentation, without forming a coherent mental model of how the system works as a whole.

In this post, I’ll try to assemble this model by examining 2 Java compilation strategies: the traditional JIT (Just-in-time) and a newer AOT (Ahead-of-Time), to understand how well each aligns with the modern deployment constraints, compare their trade-offs, limitations, and suggest ideal use cases.

My goal is to extract the core runtime differences that naturally answers many of the questions production engineers raise when deciding what to adopt.

But first, let’s step back and look at the retrospective: where we’ve been and where we are now. This context helps clarify how Java’s compilation strategies evolved and makes the underlying cause-and-effect relationships much easier to see.

Retrospective#

Java’s original compilation design#

In the mid-90s, serious server software was often a long-running application server process, that shaped the Java initial compilation design.

Native-compiled models (C/C++) were fast but usually required separate builds per OS/CPU, resulting in N-platform-specific binaries to distribute (that era had lots of serious platforms).

Interpreted models (Perl, early Python) were easier to move across platforms (shipping source code, and offering an interpreter per platform), but often too slow for hot server paths.

Given obvious drawbacks of these models, Java followed a 3-d path seen in Smalltalk-style VM systems: compile sources to portable bytecode, run it on a cross-platform VM, and use JIT compilation to turn hot bytecode into native machine code at runtime.

In that context, it was a practical trade-off: portable distribution with runtime performance, because long-lived server processes allowed runtime profiling and optimization to pay off.

New era: architectural shift#

By the early 2010s, the assumptions under which Java’s JIT strategy had worked well started to change.

Applications increasingly ran in containerized environments (Docker, ~2013; Kubernetes, ~2016) and as serverless functions (AWS Lambda, ~2014).

This shift did not make JIT unsuitable by itself, but it changed how apps were run: services now consisted of many short-lived instances that scaled up and down frequently.

In such setups, fast startup became more important than long-term peak performance, and per-instance memory usage directly affected how many instances could run and how much the infrastructure cost.

Java’s dual compilation strategy#

By the late 2010s, the JVM ecosystem began to respond to these new constraints more systematically.

The GraalVM research project gradually shifted its focus from building an alternative JIT compiler toward AOT compilation (~2018, v1.0), where it has since achieved serious, production-ready results.

At the same time, newer Java frameworks such as Micronaut (~2018) and Quarkus (~2019) emerged with AOT-native design principles, while more established frameworks like Spring Boot(~2022, v3.0) invested significant effort in adapting their internals to support AOT as an additional option.

As a result, the Java ecosystem converged toward a “dual compilation strategy”, where JIT and AOT coexist and are selected based on deployment needs rather than ideology.

Java compilation break down#

Having established the historical context, we can now break down how modern Java compilation works.

First of all we need to admit that Java does not define a formal compilation pipeline in its specifications.

Instead, the structure of the Java toolchain and JVM impl.-s naturally results in a multi-stage compilation process (the term is non-official, but reflects how the platform operates in practice).

  1. Stage #1: Source code (.java) -> bytecode (.class), occurs strictly at build time;
  2. Stage #2: Bytecode -> native code, may occur either at runtime or at build time.

The second stage is where compilation strategies diverge and has 2 alternative, mutually exclusive paths:

  • Stage #2a: (traditional, referred as JIT), since the day 1, bytecode is compiled to native code inside a running JVM (HotSpot) at runtime;
  • Stage #2b: (newer, referred as AOT), since the first release of GraalVM, bytecode is compiled to native code ahead of execution during a secondary build phase (via GraalVM compiler).

Below, each of these stages is discussed in detail.

Stage #1: sources-to-bytecode compilation#

While the JVM specifications do not mandate a particular compiler for Stage #1, OpenJDK — open-source reference impl. of the Java SE Platform — includes javac the canonical Java compiler responsible for translating Java source code into bytecode, now packaged as the jdk.compiler module inside JDK.

When a Java SE-compliant JDK distribution, such as Eclipse Temurin, BellSoft Liberica, or Amazon Corretto, is installed on a system, the javac compiler is guaranteed to be available as a command-line tool. Invoking javac command runs a thin OS-level launcher (shell script or native stub) that starts the Java compiler itself, producing JVM bytecode as output.

While the command’s generic form is fully documented, opinionated “production minimum” form is typically looks like this:

javac \
    --release <N> \
    -d <out> \
    @sources.txt

where:

  • --release <N>: compiles the code as if targeting Java <N> version;
  • -d <out>: output directory for generated .class files;
  • @sources.txt: argument file containing the list of file paths (.java source files) to compile (scales to large projects and avoids command-line length limits);

Today, javac primarily serves as a human-facing reference tool, while real-world building tools like Maven and IDEs usually drive the compiler through the JavaCompiler SPI (Service Provider Interface).

To have a broader idea what is happening under the hood during this Stage #1 compilation phase, I provide some excerpts from the official documentation.

The entire compilation process is generally divided into 3 phases:

  1. Parse & Enter: turns raw source code text into syntax trees and symbol tables;
  • Tokenization (scanning): reads raw source text and extracts tokens (keywords, identifiers, literals, operators);
  • Parsing (parsing): builds an Abstract Syntax Tree (AST) forest from the token stream according to Java grammar (one tree per .java file), to get the hierarchical structure of the code.
  • Symbol tables (entering): creates a global table of all available symbols (dictionary), conceptually a map of maps: package -> class -> members. Symbol tables exist to make name resolution fast: O(1) lookup within scopes, instead of repeated AST scans.
  1. Annotation processing: (JSR-269) runs annotation processors;
  • Recursive processing: compiler discovers annotation processors that impl. Processor interface; processors are executed in rounds, and if any processor generates new source files, those files are parsed and entered, symbol tables are extended, and another round starts until no new source files are produced.
  1. Analyse & Generate;
  • Attribution: walks the AST and resolves each name and expression to a specific symbol (from the global table) and static type, selecting the correct overloads, inferring generic types, and checking access rules, without changing the tree structure, but enriching it with “semantic information”.
  • Flow analysis: traverses method bodies using control-flow graphs to verify correct execution paths, ensuring variables are definitely assigned before use and detecting unreachable or invalid control-flow statements;
  • Generics translation: rewrites the attributed AST to eliminate generic types, applying type erasure, inserting required casts, and generating bridge methods, so the code conforms to the JVM’s non-generic type system.
  • Desugaring (lowering): rewrites high-level Java constructs into simpler forms, e.g. foreach -> for loop; try-with-resources -> try/finally or records -> final classes with an all argument constructor.
  • Bytecode generation: walks the lowered AST and emits JVM bytecode for method bodies, producing Code attributes (instructions, operand stack and local limits, exception tables), after which the class is written to a .class file.

As is shown, turning Java sources into bytecode is rather expensive, so doing it ahead of time was a smart trade-off.

This cost also scales linearly with project size: the more classes a project contains, the more work the compiler must do to produce bytecode — which further justifies paying this cost once during the build rather than at runtime.

This step completes the build phase and produces a portable artifact: Java programs are distributed as bytecode (typically packaged into *.jar archives), ready to be launched on any platform where a JVM is available.

The result is a compact, stack-based instruction set that’s easy for the JVM to reason about, letting the interpreter run a tight fetch–decode–execute loop instead of dealing with high-level language semantics at runtime.

Interestingly, Java, Kotlin, Scala, Groovy, Clojure - all JVM-based languages compile to the same JVM bytecode; the JVM does not know what language produced any given .class file.

But what next? From here, the paths diverge — with bytecode remaining the common starting point.

Stage #2a: JIT (Just-in-Time) compilation#

Before talking about Just-In-Time compilation per se, we need to look at how HotSpot VM as reference impl. of JVM executes bytecode by default.

To run a Java program, the JDK provides a command-line tool called java. Much like javac, it is a thin OS-level launcher whose sole responsibility is to start the JVM as a native process. While the form of the command is fully documented, I just provide my opinionated “baseline” often used in controlled production environments:

java \
    -Xms<size> \
    -Xmx<size> \
    -XX:[+|-]TieredCompilation \
    -jar <application>.jar,

where

  • -Xms: (in bytes) sets the initial heap size; avoiding early heap expansion and its associated overhead;
  • -Xmx: (in bytes) sets the maximum heap size; the heap will never grow beyond this limit;
  • -XX:[+|-]TieredCompilation: enables/disables the use of tiered compilation; enabled by default;
  • -jar <application>.jar: instructs the JVM to execute the specified JAR file as an application.

If not specified, HotSpot VM applies ergonomic defaults:

  • the initial heap size is set to 1/64 of physical memory;
  • the maximum heap size is capped at 1/4 of physical memory;
  • tiered compiler, using both C1 and C2 (see below).

This triggers a sequence of steps that ultimately translate bytecode into instructions understood by the underlying platform and CPU.

While the JVM specification defines what must be executed, it deliberately avoids specifying how execution is performed.

In the HotSpot VM, this freedom is realized through an adaptive execution strategy known as “tiered compilation”, which exists purely as an implementation detail.

I must admit, that tiered compilation is only partially documented: its existence and configuration are official, while the number of tiers, thresholds, and decision logic remain internal implementation details that may change across HotSpot VM versions.

To stay anchored in reality rather than speculation, the explanation below is grounded in a presentation by Oracle engineer Tobias Hartmann,- arguably the most authoritative public reference on HotSpot’s tiered compilation, see slides.

Below is a minimal sequence of steps that illustrates how execution progresses inside a HotSpot JVM after startup, focusing on how tiered compilation is gradually engaged.

  • Operating system starts HotSpot VM as native process;
  • HotSpot VM inits core runtime subsystems: class loading, interpreter, GC, etc.;
  • Entry application class (with main() method) is loaded from the local file system;
  • The main() method bytecode starts executing via interpreter instruction-by-instruction (“tier 0”);
  • When an instruction references an unknown class, JVM loads it lazily from the file system via a class loader;

From this point onward, HotSpot VM begins gathering runtime execution data (profiling), and tiered compilation may progressively come into play as methods become hot.

As bytecode continues to execute, the HotSpot VM increments lightweight runtime counters, primarily method-invocation counts and loop backedge counts. These counters serve as signals of hotness and are stored in JVM-internal runtime data structures associated with each method.

When a method or loop crosses an internal hotness threshold, HotSpot VM creates a compilation request and places it in a compiler queue. From this point, JIT compilation is initiated, although execution still continues in the interpreter until compiled code becomes available. Just-in-time compilation is typically performed asynchronously by dedicated compiler threads, allowing the app to continue running without blocking. The compiler translates the method’s bytecode, using the collected profiling data, into native machine code and installs the result into the code cache.

In HotSpot VM, this compilation may be performed by a fast, low-optimization compiler (C1, “tier 1-3”) or by a more aggressive optimizing compiler (C2, “tier 4”), depending on runtime heuristics. Intermediate C1 tiers allow HotSpot VM to balance fast compilation with progressively richer profiling before committing to expensive C2 optimization.

On subsequent invocations (or loop iterations), execution is redirected from the interpreter to the compiled code in the code cache, significantly improving throughput, because compiled native code runs directly on the CPU without the interpreter’s extra instruction-by-instruction handling.

Because compiled code is based on speculative assumptions about runtime behavior, HotSpot VM may invalidate and remove affected entries from the code cache when those assumptions no longer hold. Execution then safely resumes in the interpreter via a process known as deoptimization.

The visualization of this runtime mechanism is present on Fig. 2 (present for a case when the link to the Tobias Hartmann slides is no longer exists).

As is shown, HotSpot’s JIT execution model starts slower to gather runtime profiling data, then accelerates as hot code is compiled, typically reaching peak throughput once code is optimized by C2 JIT compiler, with occasional regressions due to deoptimization.

This behavior reflects a commonly observed property of real programs: execution is highly skewed - a small set of hot methods and loops dominates runtime. HotSpot’s tiered compilation mirrors this Pareto-like rule by focusing optimization on hot code, while cold paths remain interpreted or are never loaded into the JVM at all.

Finally, Oracle designs tiered compilation to be used with its default settings for most workloads. While official documentation explains the mechanism and its rationale, it provides no general performance-tuning guidance despite exposing several related JVM flags. Disabling tiered compilation forces HotSpot VM into a non-tiered execution mode, whose exact behavior is an implementation detail and may vary across JVM versions. For a practice-oriented discussion of edge-case tunings, see this Alibaba Cloud Community blog post.

Stage #2b: AOT (Ahead-of-Time) compilation#

Ahead-of-time (AOT) compilation follows a fundamentally different — almost orthogonal — idea compared to Just-in-time (JIT). Instead of compiling code incrementally at runtime, the goal is to produce a fully native, platform-specific executable ahead of execution, much like traditional C/C++ binaries.

In this model, everything the program is expected to use at runtime must be known and compiled in advance. This differs from the traditional JVM execution model, which relies on a dynamic runtime and runtime-based optimizations. Thus, AOT compilation has been foreign to the Java platform and is still an area where the ecosystem is actively catching up.

Historically, most experimental and alternative compilation models in the Java ecosystem have been developed by Oracle Labs under the GraalVM umbrella, and currently both open-source GraalVM CE (Community Edition) and Oracle GraalVM commercial enterprise platform co-exists.

While this experimental platform offers a wide range of capabilities to explore, this chapter focuses first on its Ahead-of-time (AOT) compilation support.

A GraalVM JDK includes a utility called native-image, available in the \bin directory of the GraalVM installation. Conceptually, it plays a role similar to java utility: a thin OS-level launcher that starts a JVM — not to run the application, but to analyze, compile, and link it into a native binary for a specific operating system and CPU architecture.

Full set of arguments and params is listed in the official GraalVM CE documentation, with the warnings that the available options and defaults may vary across versions. Below I provide my opinionated, generic baseline form for this command:

native-image \
    -jar <app-name>.jar \
    -o <app-native-name> \
    -H:MaxHeapSize=<? | 512m> \
    -march=<native | compatibility> \
    --gc=<G1 | serial | epsilon> \
    -O<level>

where

  • -jar <app-name>.jar: the target bytecode to be native compiled;
  • -o <app-native-name>: the resulting name of the native executable;
  • -H:MaxHeapSize=<? | 512m>: the max heap size; defaults to 80% of available RAM on the target machine;
  • -march=<native | compatibility>: machine architecture; generate instructions for a specific machine type: native for best performance, compatibility for best portability;
  • --gc=<value>: select a native image garbage collector impl.; serial is default one, G1 is only available in Oracle GraalVM;
  • -O<level>: control code optimizations; defaults to 2 - aims for good performance at a reasonable file size.

As it shown, the tool accepts the Java bytecode per se, as well as a set of flags to tune the level of image optimisation, heap size, GC choice, machine architecture etc.

This process triggers a sequence of steps that ultimately translate bytecode into a native executable, ready to run directly on the operating system — without requiring a JVM at runtime.

Let’s walk through them.

First, the native-image launcher starts a HotSpot JVM, and the entire image-building process runs inside that hosted JVM environment. All work happens there before the final binary is produced.

Christian Wimmer’s presentation outlines the core phases of native image generation:

  • Points-to Analysis;
  • Run Initializations;
  • Heap Snapshotting.

Let’s briefly look at what each step does.

The first phase performs whole-program static analysis to determine which classes, methods, and fields are actually reachable from the application’s entry points. It constructs a call graph and object graph, iteratively expanding them until no new reachable elements are discovered (the fixed point). Only reachable code is included in the final image.

In the second phase, selected classes are initialized at build time rather than at runtime. Their static initializers execute during image generation, which may create objects or trigger additional reachability. This can expand the analysis graph and feed back into the previous step.

In the last phase, objects created during build-time execution are captured into an image heap. This heap snapshot is serialized into the binary’s .data section. As a result, when the executable starts, it already contains a preconstructed object graph, enabling fast startup.

He emphasizes that the process is iterative: analysis and initialization repeatedly influence each other. Each cycle may reveal new reachable elements, and the process continues until a fixed point is reached — when no further changes occur.

In addition to the application code and dependencies, the native-image tool also takes Substrate VM as an input - a lightweight runtime implementation that provides essential services such as memory management, threading, and garbage collection for the native executable. It is analyzed and compiled together with the app code to form the final executable.

At runtime, Substrate VM executes native machine code in much the same way HotSpot executes native code from the code cache — but the code’s origin, lifetime, and mutability are completely different.

Substrate VM executes ahead-of-time compiled machine code embedded in the binary, whereas HotSpot executes JIT-compiled code stored in its code cache.

Native executable starts fast (hundreds of ms for cold starts, tens ms for warm starts) because class loading, bytecode verification, JIT initialization, and much of framework bootstrapping are shifted to build time. The executable already contains compiled machine code and a preconstructed heap, so the process begins executing application logic almost instantly.

Native executable typically has a smaller memory footprint (tens of Mb as RSS (Resident Set Size)) because it eliminates the JVM’s JIT infrastructure, profiling data, and large metadata structures. Only reachable classes and minimal runtime support are included in the binary. As a result, both startup memory and steady-state memory are often significantly lower.

Horizontal scalability often improves with native images because each instance consumes less memory and has lower runtime overhead, allowing more replicas to run on the same node. Faster startup also improves elasticity and auto-scaling responsiveness. The benefit comes from higher instance density rather than necessarily higher per-instance throughput.

However, these benefits come with structural trade-offs: AOT relies on a closed-world assumption. All reachable code must be known at build time. As a result, dynamic features such as unrestricted reflection, runtime class loading, and bytecode generation are either restricted or must be declared explicitly during image generation.

Overall, AOT compilation shifts complexity from runtime to build time. The result is a lean, predictable executable optimized for fast startup and high deployment density — at the cost of reduced runtime adaptability and dynamic features support.

Conclusion#

Having outlined the underlying ideas of both compilation strategies, it’s time to summarize advantages and disadvantages of each one and actually answer main questions that arise when deciding what to adopt in practise.

The key advantages of JIT compilation:

  • Best peak performance;
  • Fast iteration during development;
  • Full support for runtime features (reflection, classloading, dynamic proxies).

JIT advantages make it best for:

  • Large domain-heavy monolithic applications;
  • Long-lived services with stable lifecycles.

The key advantages of AOT compilation:

  • Near-instant cold start;
  • Small and predictable memory footprint;
  • Stable and consistent throughput from the start (no warm-up phase).

AOT advantages make it best for:

  • Tiny containerized microservices with H-scaling scenarios;
  • Serverless functions (pay-per-invocation);
  • CLI tools and short-lived utilities.

But if AOT is so promising, should it be default choice?

Not necessarily. Most Java workloads are still long-running services where adaptive JIT optimization delivers better sustained throughput and architectural flexibility. When considering AOT, its documented trade-offs should be acknowledged. As noted in the BellSoft analysis:

  • Build resources. Native image generation is RAM- and *CPU-*intensive, often requiring several Gb of RAM — a non-trivial cost for CI/CD pipelines.
  • No optimization. Without a JVM and profiling infrastructure, the executable cannot recompile hot paths at runtime based on real production behavior.
  • GC choice. Compared to HotSpot’s rich GC landscape, native images currently offer a narrower selection (e.g., SerialGC in GraalVM CE, G1 in EE).

Another question is why to use Java stack for native, if other platforms designed with AOT in mind? Indeed, platforms like Go, Rust were designed with AOT-first in mind and provide even faster compilation, lighter-weight binaries and memory footprint. However, Java AOT allows teams already in the JVM ecosystem to meet deployment constraints (cold start, memory limits) without abandoning language familiarity, tooling, libraries, architectural investment. The choice is therefore not about performance absolutism, but about ecosystem continuity.

To sum up, Java’s real strength is not JIT or AOT in isolation, but the ability to choose between them without changing the language or ecosystem. The same Java codebase can run on a dynamically optimizing JVM or be compiled ahead-of-time into a native executable, depending on deployment constraints.

Only a small number of modern platforms (notably Java and .NET) offer this level of execution-model flexibility within a single, mature ecosystem — and that optionality is what makes the Java stack uniquely adaptable across radically different workloads.
