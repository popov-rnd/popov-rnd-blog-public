RabbitMQ vs ActiveMQ Artemis benchmark

"In this benchmark, we compare the 2 message brokers throughput and latency characteristics with OMB framework"

A few weeks ago, I came across a ResearchGate paper by Rokin Maharjan et al. that compared the performance of 4 messaging systems, based on OpenMessaging Benchmark (OMB) framework.

While the results for Kafka and Redis aligned with common expectations, the numbers for RabbitMQ and Apache Artemis made me pause.

In their benchmark, Rabbit achieved higher throughput, while Artemis showed lower end-to-end latency.

To better understand these results, I decided to run my own focused benchmark, limiting the scope to Rabbit and Artemis only.

The goal was not to challenge the paper’s conclusions, but to fill in the gaps it left. In particular, the article provided little detail about the experimental conditions under which the benchmarks were run, e.g. whether publisher confirms were enabled, whether message durability was turned on, etc.

These details are critical for interpreting the results, yet they were not discussed.

Furthermore, the benchmarks themselves were conducted more than two years ago (in 2023), making it valuable to revisit the comparison with a fresh setup and a deeper focus on behavior and interpretation rather than just outcomes.

Brokers#

At the start, I had very high-level understanding about internal architectures of the systems being compared: RabbitMQ and Apache Artemis.

What I knew was:

  • that RabbitMQ is an old beast originated from around 2007, written in rather rare programming language Erlang, known for its actor-like concurrency model based on lightweight processes and message passing. It is strongly protocol-driven, with AMQP 0-9-1 defining its core semantics, and has evolved continuously while remaining actively developed to this day.
  • that Apache Artemis is a complete re-write of ActiveMQ broker, using modern Java 8+ concurrency, efficient event loops backed by Netty at transport layer. It supports the JMS specification and, as a result, can expose legacy enterprise semantics such as XA / two-phase commit transactions.

Given this limited architectural context, the benchmark was as a starting point — a way to move beyond assumptions and develop a more accurate mental model of how these systems behave under comparable conditions, and where their real strengths and trade-offs lie.

Method#

The OMB framework itself is reviewed separately in this blog post, so I will not repeat its internal details here.

Although OMB supports benchmarking in distributed environments, for this initial experiment I deliberately chose a local-only setup to reduce complexity and isolate broker behavior as much as possible.

Both target systems — RabbitMQ and Apache Artemis — were deployed as Docker containers. The OMB framework itself was compiled and executed directly on the host machine.

For each benchmark run the target broker container was recreated from scratch. Only one broker container was running at any given time (never both simultaneously).

OMB was configured to execute each workload for 5 minutes, with an additional 1-minute warm-up period to allow the system to reach a steady state before measurements were recorded.

To monitor resource usage during benchmark execution, I used Docker’s native runtime metrics via the docker stats <container-name> command, primarily to observe CPU and RAM utilization trends throughout each run.

OMB reports publish throughput as a time series of per-interval publishRate samples. For comparison, in this study I reported the average sustained publish rate computed across the steady-state window.

Hardware#

All experiments were conducted on my desktop Linux Mint PC, with the following configuration:

  • CPU Intel Core i7, 8x2.9GHz
  • SSD NVMe PCIe 4.0 x4, 500GB, write up to 5Gb/sec
  • RAM DDR4-2666, 32Gb, up to 21.3 Gb/sec

Software#

On my desktop PC I used the following software:

  • OS Linux Mint (kernel 5.15.0)
  • Docker engine (v. 29.1.3)

I used the latest docker images available for both target systems:

  • rabbitmq:4.2.2
  • apache/activemq-artemis:2.44.0

For OMB framework I updated the official Java clients for both target systems:

  • amqp-client v. 5.28.0 (dec. 2025)
  • artemis-core-client v. 2.44.0 (nov. 2025

Also, I compiled and ran the OMB process with JDK 25 Temurin. I was launching OMB via its native bash scripts, supplied with the source code itself.

Scenarios#

Overall I was able to benchmark around 3 different topologies:

  • Basic: 1 producer/ 1 topic/ 1 queue/ 1 consumer;
  • Independent queues: 8 producers/ 8 topics/ 8 queues/ 8 consumers;
  • Fan-out: 8 producers/ 1 topic/ 8 queues/ 8 consumers.

The OMB framework created the required topologies automatically at startup based on the provided YAML configuration.

For all experiments I used single broker deployment, classic queues for RabbitMQ and default queues for Apache Artemis.

I played around number 8 to match the number of physical CPU cores available on my PC (+8 cores with hyper-threading). The assumption was that this would reveal how well the systems parallelize job across cores.

I also attempted to evaluate scenarios with multiple producers and consumers interacting with a single queue. While OMB itself supports this topology, the available broker drivers differ in their capabilities: the Rabbit’s OMB driver allows this configuration, whereas the Apache Artemis’ OMB driver doesn’t support it out of the box.

To keep scenarios comparable across both systems, this topology was therefore excluded from the benchmark.

Results#

In the first benchmark (Fig. 1), I compared max throughput - the max publish rate, for different payload sizes. Both systems compete hard with no clear winner in all payloads sizes. The bigger message size - the less messages a system can ingest per sec. For non-durable messages Artemis showed +~25% better max throughput than Rabbit.

Secondly (Fig. 2, log scale!), I compared publish latencies at p99 with strictly fixed publish rate. For this experiment I selected 1Kb payload as one of the most common in real production systems. Rabbit confirms pubs much faster (sub. milliseconds) than Artemis (single digit milliseconds). Interesting, that Artemis achieves the same sub. milliseconds publish latency for non-durable messages.

Lastly (Fig. 3), I compared consumers’ throughput for 3 different topologies (described above). The chart reveals that Artemis can better parallelize work and achieve more efficient internal routing in scenarios with multiple queues.

For all experiments, Rabbit’s docker containers utilized much less RAM (100-150Mb up to 300Mb) then Artemis (300-700Mb up to 1.3Gb), but at the same time more intensively used CPU (~+25-50%). Additionally, Artemis demonstrated a more even CPU utilization across cores.

Interpretation#

First, the main question: “what prevents message brokers from ingesting more messages per sec? What is the first bottleneck they face, that architecturally plateaus throughput?”

During my experiments I observed that:

  • CPU cores never loaded at 100%;
  • Network card limits are not relevant for local set-up;
  • RAM consumption stayed far less than was available in my system (up to 1.3Gb vs 32Gb available);
  • SSD NVMe being able to do 5 GB/sec write bandwidth, never saturated (used 0.1% of what is capable).

This indicates that hardware is not the first limiting factor in messaging systems.

Unfortunately, most broker docs focus on what can be tuned rather than why a throughput ceiling exists. To understand the latter, it is useful to look at the flow-control mechanisms described by the brokers themselves:

  • RabbitMQ: Flow control.
  • Apache Artemis: Producer flow control;

Both brokers explicitly implement producer-side flow control. Rather than pushing the system until hardware resources are exhausted, they continuously assess whether the broker can safely accept additional messages and deliberately throttle producers to preserve internal stability and correctness.

Importantly, neither broker docs define a precise threshold, formula, or algorithm that determines when flow control is applied. The decision is based on internal coordination and safety constraints, not on a fixed resource limit that can be calculated in advance.

As a result, what is ultimately measured in benchmarks is not raw hardware capacity, but the maximum rate at which the broker can repeatedly complete a full accept -> coordinate -> acknowledge cycle under the current runtime conditions.

Another natural question is: why brokers activate these protective mechanisms long before any visible hardware exhaustion occurs?

The answer lies in the messaging specifications they implement, namely:

  • AMQP;
  • Jakarta Messaging (JMS).

These standards define the classic messaging model, which mandates ordered message acceptance and explicit acknowledgements within a queue or session. This ordering guarantee is not a best-effort optimization; it is a correctness requirement.

To preserve these guarantees, brokers must coordinate message acceptance in a strictly ordered and internally synchronized way before messages are persisted or delivered.

Internally, this means the broker protects a small shared piece of state, so only one message at a time is allowed to move forward, similar to a locked or synchronized critical section in code.

This behavior can be understood through Amdahl’s Law, which states that the presence of a mandatory serial execution path places an upper bound on achievable performance.

As a result, throttling occurs based on how fast this synchronized acceptance and acknowledgement path can progress (to avoid runaway in-flight state and memory growth), not on raw CPU, SSD, or network utilization.

Secondly, why as messages grow in size, msg/sec throughput drops?

The above-mentioned accept → coordinate -> acknowledge cycle has a largely fixed coordination overhead per message, but message size increases the work performed inside that cycle. Specifically, larger messages consume more producer credits (byte-based flow control), increase copying and buffer-management costs, thus extending the time a message spends in the ordered acceptance path.

Even if total bandwidth remains relatively low, each message holds the serialized acceptance path for longer, reducing the rate at which the next message can advance.

As a result, fewer messages per second can pass through the same ordered coordination point.

Lastly, why Apache Artemis publish latency is so huge (single digit ms) compared to RabbitMQ (sub. millisecond) for durable messages?

This difference is primarily explained by how the two brokers implement durable message publish confirmation semantics.

For a single-node broker, publish latency for durable messages is dominated by exactly 2 mechanisms — regardless of vendor:

  • write into OS page cache (typically measured in µs), which guarantees durability against broker process crashes;
  • sync into stable storage fsync (typically measured in ms), which guarantees durability against full power failures (similar to ACID in RDBMS).

There is no third magic source of durability latency.

In Apache Artemis, a durable publish is not acknowledged until the message has been safely recorded in the broker’s journal.

Furthermore, the broker-wide journal-datasync setting (enabled by default) determines whether Apache Artemis waits for fsync to complete before acknowledging a journal write. Hence, default behavior is to perform fsync before actual publish confirmation, hence latency in milliseconds.

By contrast, Rabbit takes a more relaxed approach to durability acknowledgements by batching disk writes more aggressively, decouple publisher confirms from immediate fsync in default configuration for classic queues.

This blog post by Rabbit’s maintainer, explicitly discusses the risks, suggesting that for stronger guarantees, one can use quorum queues.

This means that, in practice, Rabbit confirms publishes earlier, while persistence continues in the background or is shared across a batch of messages.

In other words, Artemis trades higher publish latency for stricter, more explicit durability semantics, whereas Rabbit trades some immediacy in durability confirmation for lower publish latency.

The observed benchmark results are therefore a direct reflection of these design trade-offs, not an anomaly or tuning issue.

In a follow-up series, I plan to examine high-availability (HA) configurations for both brokers and how these additional coordination and replication requirements further affect throughput and latency.
